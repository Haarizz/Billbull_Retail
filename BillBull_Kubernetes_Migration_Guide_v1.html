1. Gaps this guide fixes vs. a generic k8s doc

A generic "install k3s, apply some YAML" guide misses things that will actually break this app. This guide addresses them directly:

GapWhy it matters here
Product/document images on local diskupload.path=uploads/ is per-client local disk. A naive PVC + multi-replica Deployment loses files or corrupts writes. Addressed in Section 3.
Playwright/Chromium PDF engineNeeds specific base-image deps and a bigger /dev/shm than k8s's 64MB default, or invoice/report PDF generation crashes silently. Addressed in Section 2 and the Deployment spec in Section 5.
12 near-identical client deploymentsHand-writing 12×(Deployment+Service+Ingress+Secret) doesn't scale. Templated with Helm from the start — Section 7.
External Postgres reachabilityPostgres is not containerized; pods must reach the host's existing DB, not a k8s-internal one. Addressed in Section 5.
ddl-auto=update + rolling deploysOverlapping old/new pod versions during a rolling update is riskier than today's single-instance systemd restart. Mitigated with replicas: 1 + strategy: Recreate, matching current behavior exactly.

2. Containerize: Dockerfiles

One-time Build once per app, reuse the same image for every client (config, not code, differs between clients).

Backend — billbull-backend/Dockerfile

FROM maven:3.9-eclipse-temurin-17 AS build
WORKDIR /app
COPY pom.xml .
RUN mvn -o -B dependency:go-offline || mvn -B dependency:go-offline
COPY src ./src
RUN mvn -B clean package -DskipTests

FROM eclipse-temurin:17-jre-jammy
WORKDIR /app

# Playwright/Chromium needs these system libs on a bare JRE image
RUN apt-get update && apt-get install -y --no-install-recommends \
    libnss3 libatk-bridge2.0-0 libx11-xcb1 libxcomposite1 libxdamage1 \
    libxrandr2 libgbm1 libasound2 libpangocairo-1.0-0 libgtk-3-0 fonts-liberation \
    && rm -rf /var/lib/apt/lists/*

COPY --from=build /app/target/billbull-backend-*.jar app.jar

ENV PLAYWRIGHT_BROWSERS_PATH=/opt/ms-playwright
RUN java -cp app.jar com.microsoft.playwright.CLI install --with-deps chromium

EXPOSE 8080
ENTRYPOINT ["java", "-XX:MaxRAMPercentage=75.0", "-jar", "app.jar"]
Playwright install moves from a manual per-server step (see the old guide's "Playwright / PDF Engine Note") into the image build, once. No client deploy ever needs to remember it again.

Frontend — billbull-frontend/Dockerfile

FROM node:20-alpine AS build
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

FROM nginx:1.27-alpine
COPY --from=build /app/dist /usr/share/nginx/html
COPY nginx.conf /etc/nginx/conf.d/default.conf
EXPOSE 80

nginx.conf is the same SPA-fallback config you'd use today — serve static files, fall back to index.html for client-side routing:

server {
  listen 80;
  root /usr/share/nginx/html;
  index index.html;
  location / { try_files $uri $uri/ /index.html; }
}

Build and push

docker build -t ghcr.io/billbull/backend:2026.07.16 billbull-backend/
docker build -t ghcr.io/billbull/frontend:2026.07.16 billbull-frontend/

docker push ghcr.io/billbull/backend:2026.07.16
docker push ghcr.io/billbull/frontend:2026.07.16
Use GitHub Container Registry (or another private registry) from day one, not k3s ctr images import. The import approach means every rebuild requires manual copying to the server — no real CI/CD, and it doesn't scale to two servers.

3. Decide the /uploads strategy

This is the one decision that affects every client manifest below. Two options:

Option A: PVC (local-path), no code changeOption B: Object storage (MinIO/S3), code change
EffortLow — infra onlyMedium — swap FileUploadUtil to write/read via S3 client
ConstraintMust pin replicas: 1 + strategy: Recreate per client backend foreverNone — any replica count, any node
Node mobilityPod tied to whichever node holds the PVC's local diskAny node, any cluster
Matches today's behaviorYes — you already run exactly 1 backend instance per clientImproves on today
Recommendation: start with Option A. It's a direct lift of what you run today (one instance per client, brief downtime on redeploy, same as a systemctl restart) and requires zero backend code changes. Move to Option B (MinIO) as a follow-up phase once the cluster itself is proven, not as a blocker to migrating. The worked example below uses Option A; the manifest is written so swapping to Option B later only means dropping the PersistentVolumeClaim/volumeMounts block, not touching anything else.

4. Install k3s + cert-manager on one server

One-time per server Do this on 62.72.59.119 first (8 clients). Repeat later on 77.37.49.42 (see Section 9 for whether that becomes a second node or a second cluster).

curl -sfL https://get.k3s.io | sh -
sudo systemctl status k3s

mkdir -p ~/.kube
sudo cp /etc/rancher/k3s/k3s.yaml ~/.kube/config
sudo chown $(id -u):$(id -g) ~/.kube/config
kubectl get nodes

# cert-manager for automatic Let's Encrypt certs (replaces certbot)
kubectl apply -f https://github.com/cert-manager/cert-manager/releases/latest/download/cert-manager.yaml
kubectl get pods -n cert-manager --watch

Create a ClusterIssuer (one-time, cluster-wide):

cat <<'EOF' | kubectl apply -f -
apiVersion: cert-manager.io/v1
kind: ClusterIssuer
metadata:
  name: letsencrypt-prod
spec:
  acme:
    server: https://acme-v02.api.letsencrypt.org/directory
    email: ae@geebu.io
    privateKeySecretRef:
      name: letsencrypt-prod-key
    solvers:
      - http01:
          ingress:
            class: traefik
EOF

k3s ships with Traefik as its built-in Ingress controller — no separate nginx-ingress install needed. It listens on ports 80/443, same as nginx does today.

5. Worked example: migrate geebu end-to-end

Per client, first time Chosen because it's a real, low-traffic client, currently on port 8088 / server 62.72.59.119. Every step below is geebu-specific; Section 7 turns this into a template for the other 11.

1Namespace

kubectl create namespace billbull

All clients share one namespace (matches today's model — all clients already share one server/nginx instance). Use per-client namespaces only if you later want hard RBAC isolation between clients.

2Secret — DB credentials + JWT secret

Maps directly from today's systemd unit's Environment= lines for billbull-geebu:

kubectl create secret generic geebu-backend-secret \
  --namespace billbull \
  --from-literal=SPRING_DATASOURCE_URL='jdbc:postgresql://localhost:5432/geebu_db' \
  --from-literal=SPRING_DATASOURCE_USERNAME='postgres' \
  --from-literal=SPRING_DATASOURCE_PASSWORD='<actual-password>' \
  --from-literal=JWT_SECRET='<actual-jwt-secret>'
Substitute the real values from the current /etc/systemd/system/billbull-geebu.service Environment= lines on 62.72.59.119. Never commit this command with real values to Git — run it directly on the server, or generate the Secret YAML into a file that's gitignored.

Postgres is not in the cluster — it stays exactly where it is today. localhost:5432 from the old systemd unit becomes the server's real LAN/host IP from inside the pod (k3s pods don't share the host network namespace). Find it with ip addr on the host, or use the k3s host-gateway alias:

# if Postgres listens on the host's normal interface, use that IP directly, e.g.:
SPRING_DATASOURCE_URL=jdbc:postgresql://10.0.0.5:5432/geebu_db

# or, if you want a portable "the host machine" reference, add to the Deployment pod spec:
#   hostAliases:
#     - ip: ""
#       hostnames: ["dbhost"]
# then use jdbc:postgresql://dbhost:5432/geebu_db

Also confirm Postgres's pg_hba.conf/listen_addresses accepts connections from the pod network CIDR (k3s default is 10.42.0.0/16), not just 127.0.0.1.

3ConfigMap — non-secret config

cat <<'EOF' | kubectl apply -f -
apiVersion: v1
kind: ConfigMap
metadata:
  name: geebu-backend-config
  namespace: billbull
data:
  SPRING_PROFILES_ACTIVE: "geebu"
  SERVER_PORT: "8080"
EOF

4PersistentVolumeClaim for /uploads (Option A from Section 3)

cat <<'EOF' | kubectl apply -f -
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: geebu-uploads-pvc
  namespace: billbull
spec:
  accessModes: ["ReadWriteOnce"]
  storageClassName: local-path
  resources:
    requests:
      storage: 10Gi
EOF
Before first boot, copy the existing files across so the client doesn't lose history: kubectl cp /root/Billbull/billbull-backend/uploads/ billbull/<geebu-pod>:/app/uploads/ (run after the pod exists once, or mount the PVC via a throwaway pod first and rsync into it).

5Deployment

cat <<'EOF' | kubectl apply -f -
apiVersion: apps/v1
kind: Deployment
metadata:
  name: geebu-backend
  namespace: billbull
spec:
  replicas: 1
  strategy:
    type: Recreate        # matches today's single-instance systemctl restart; no split-brain on the PVC
  selector:
    matchLabels: { app: geebu-backend }
  template:
    metadata:
      labels: { app: geebu-backend }
    spec:
      containers:
        - name: backend
          image: ghcr.io/billbull/backend:2026.07.16
          ports: [{ containerPort: 8080 }]
          envFrom:
            - configMapRef: { name: geebu-backend-config }
            - secretRef: { name: geebu-backend-secret }
          resources:
            requests: { cpu: "250m", memory: "512Mi" }
            limits: { cpu: "1", memory: "1Gi" }
          volumeMounts:
            - name: uploads
              mountPath: /app/uploads
            - name: dshm
              mountPath: /dev/shm
          readinessProbe:
            httpGet: { path: /actuator/health, port: 8080 }
            initialDelaySeconds: 30
            periodSeconds: 10
      volumes:
        - name: uploads
          persistentVolumeClaim: { claimName: geebu-uploads-pvc }
        - name: dshm
          emptyDir: { medium: Memory, sizeLimit: 512Mi }   # Chromium/Playwright needs more than k8s's 64MB default
EOF
If Spring Boot Actuator's /actuator/health isn't exposed in this app, swap the readinessProbe for a tcpSocket: { port: 8080 } check instead — confirm which is available before applying.

6Service

cat <<'EOF' | kubectl apply -f -
apiVersion: v1
kind: Service
metadata:
  name: geebu-backend
  namespace: billbull
spec:
  selector: { app: geebu-backend }
  ports: [{ port: 8080, targetPort: 8080 }]
EOF

7Frontend (shared, deploy once — not per client)

cat <<'EOF' | kubectl apply -f -
apiVersion: apps/v1
kind: Deployment
metadata:
  name: billbull-frontend
  namespace: billbull
spec:
  replicas: 2
  selector: { matchLabels: { app: billbull-frontend } }
  template:
    metadata: { labels: { app: billbull-frontend } }
    spec:
      containers:
        - name: frontend
          image: ghcr.io/billbull/frontend:2026.07.16
          ports: [{ containerPort: 80 }]
---
apiVersion: v1
kind: Service
metadata:
  name: billbull-frontend
  namespace: billbull
spec:
  selector: { app: billbull-frontend }
  ports: [{ port: 80, targetPort: 80 }]
EOF

This one frontend Deployment/Service is reused by every client's Ingress below — same as today, where every client hits the same /var/www/billbull/dist build.

8Ingress — routes the domain to frontend, /api to backend

cat <<'EOF' | kubectl apply -f -
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: geebu-ingress
  namespace: billbull
  annotations:
    cert-manager.io/cluster-issuer: letsencrypt-prod
spec:
  tls:
    - hosts: ["geebu.billbull.app"]
      secretName: geebu-tls
  rules:
    - host: geebu.billbull.app
      http:
        paths:
          - path: /api
            pathType: Prefix
            backend:
              service: { name: geebu-backend, port: { number: 8080 } }
          - path: /uploads
            pathType: Prefix
            backend:
              service: { name: geebu-backend, port: { number: 8080 } }
          - path: /
            pathType: Prefix
            backend:
              service: { name: billbull-frontend, port: { number: 80 } }
EOF
Adjust the path list to whatever the frontend's axiosConfig.js actually calls (likely /api/** and possibly others like /uploads/**, /ws/** if used) — check VITE_API_BASE_URL handling and confirm no other backend-served paths are missing before cutover.

DNS: point geebu.billbull.app at the k3s server's IP (same IP it points at today if you're keeping it on 62.72.59.119) — cert-manager will provision the TLS cert automatically via HTTP-01 once Traefik is reachable on that domain.

6. Verify the migrated client

kubectl get pods -n billbull -l app=geebu-backend --watch
kubectl logs -n billbull deploy/geebu-backend -f
kubectl get certificate -n billbull geebu-tls
kubectl describe ingress geebu-ingress -n billbull

Functional checklist before pointing DNS at it for real (or before disabling the old systemd unit):

☐App loads at https://geebu.billbull.app, cert is valid
☐Login works (JWT issuance/validation — confirms JWT_SECRET Secret is correct)
☐Existing data visible (confirms DB connectivity to the external Postgres)
☐Invoice/report PDF generates (confirms Playwright/Chromium + /dev/shm sizing)
☐Product image upload + display works (confirms PVC mount + copied files)
☐Flyway ran clean on boot (kubectl logs shows migration success, no baseline errors like the earlier Hilite incident)
Keep the old billbull-geebu systemd service running (just stop pointing DNS/traffic at it) until this checklist passes. Don't decommission it same-day.

7. Template it: Helm chart for the rest

One-time Once geebu is verified, convert the manifests above into a Helm chart with per-client values, instead of hand-writing this 11 more times.

billbull-chart/
  Chart.yaml
  templates/
    secret.yaml
    configmap.yaml
    pvc.yaml
    deployment.yaml
    service.yaml
    ingress.yaml
  values-geebu.yaml
  values-royaltools.yaml
  values-qa.yaml
  ... one values file per client

values-geebu.yaml (the only file that changes per client):

client: geebu
domain: geebu.billbull.app
springProfile: geebu
dbUrl: jdbc:postgresql://10.0.0.5:5432/geebu_db
uploadsSizeGi: 10

Install/upgrade a client:

helm upgrade --install geebu ./billbull-chart -n billbull -f values-geebu.yaml \
  --set-string secrets.dbPassword=$GEEBU_DB_PASS \
  --set-string secrets.jwtSecret=$GEEBU_JWT_SECRET

New backend image release, all clients at once:

for f in values-*.yaml; do
  client=$(basename "$f" .yaml | sed 's/values-//')
  helm upgrade "$client" ./billbull-chart -n billbull -f "$f" \
    --reuse-values --set image.tag=2026.07.20
done

8. Roll out remaining clients & retire systemd

Go client-by-client, not big-bang. For each:

  1. helm install with that client's values (Secret populated from its current systemd unit's env vars)
  2. Copy its /uploads folder into the new PVC
  3. Run the verification checklist against a temporary hostname or by editing /etc/hosts locally before flipping real DNS
  4. Flip DNS for that client's domain to the k3s server
  5. Watch logs/traffic for 24–48h
  6. Only then: systemctl stop billbull-<client> && systemctl disable billbull-<client> on the old server
Do not batch-disable multiple systemd services in one pass, even after everything "looks fine" — each client is an independent DNS cutover with its own blast radius.

9. Two-server topology decision

62.72.59.119 (8 clients) and 77.37.49.42 (4 clients) are physically separate today. Two options once both are on k3s:

A: Two independent single-node clustersB: One multi-node cluster (join 77 as a k3s agent to 62)
Setupcurl ... | sh - on each, independentlycurl ... | K3S_URL=https://62...:6443 K3S_TOKEN=<token> sh - on 77
Uploads PVC (Option A)Safe — pod always lands on the node whose local disk actually holds that client's filesUnsafe without care — scheduler could place a client's pod on the wrong node, away from its PVC's disk, unless pinned with nodeSelector/node affinity per client
Shared Ingress/DNS mgmtTwo separate configs to maintainOne Ingress config, easier long-term ops
Recommendation: keep them as two independent single-node clusters (Option A) for now — it exactly matches today's physical/data boundary and needs no extra pinning logic. Revisit joining them into one cluster only after /uploads moves to shared object storage (Section 3, Option B), at which point node placement no longer matters.
