Basic Part 1 — What is a container, really?

The problem containers solve

Before containers, "it works on my machine" was a real, constant problem. Your Java app needs a specific JDK version, specific system libraries (like the ones Playwright/Chromium need — libnss3, libgbm1, etc.), a specific working directory layout. The server it runs on needs all of that to match, exactly, or it breaks in ways that are hard to reproduce.

Think of a container like a shipping container in the real world: it doesn't matter what's inside (furniture, electronics, food) — any ship, truck, or crane can move it because the outside shape and interface are standardized. A software container does the same thing: it packages your app plus everything it needs to run, in a standard shape that any machine with a container runtime can execute identically.

Container vs. Virtual Machine — the key difference

You may have used VMs (VirtualBox, VMware) before. A VM virtualizes an entire computer — its own kernel, its own OS, booting takes minutes, uses gigabytes of RAM just for the OS. A container shares the host machine's kernel and only isolates the application layer — processes, filesystem, network — so it starts in milliseconds and uses only the memory the app itself needs.

Virtual MachineContainer
What's virtualizedEntire hardware + OS kernelJust the process's view of the filesystem/network — shares host's kernel
Startup timeMinutesMilliseconds to seconds
SizeGigabytes (full OS)Megabytes to low-hundreds-MB (just your app + deps)
What BillBull uses it for—Packaging the Spring Boot backend + React frontend, exactly what we did

What's actually inside a running container

When our billbull-backend:pilot container runs, it is one Linux process (the JVM) on the host machine (77.37.49.42) — but that process believes it has its own private filesystem (starting from /app), its own process list (it can't see nginx or Postgres running alongside it), and its own network stack. This isolation is built from Linux kernel features called namespaces (isolate what a process can see) and cgroups (limit how much CPU/memory it can use — this is exactly what our resources.limits in deployment.yaml configures).

This is why, inside the container, upload.path=uploads/ resolves to /app/uploads — /app is the container's own private root-relative working directory, completely separate from anything at /app on the actual host server.

Basic Part 2 — Docker: images, layers, the Dockerfile

Image vs. container — the distinction that trips everyone up

An image is a read-only template — a frozen snapshot of a filesystem plus metadata about how to run it (what command to execute, what port it listens on). A container is a running (or stopped) instance of that image — the same relationship as a class and an object in programming, or a recipe and a cooked meal.

docker build -t billbull-backend:pilot .     # builds the IMAGE (the recipe)
docker run billbull-backend:pilot            # runs a CONTAINER (an instance of it)

You can run the same image many times, getting independent containers each time — this is exactly how our Deployment's replicas: 2 for the frontend works: two separate running containers, both instances of the one billbull-frontend:pilot image.

Layers — why builds get faster after the first one

A Dockerfile builds an image as a stack of layers, one per instruction. Each layer is cached. If a layer's inputs haven't changed, Docker reuses the cached result instead of re-running it. This is exactly why our real backend build looked like this the second time:

=> CACHED [build 3/7] COPY pom.xml .                    0.0s
=> CACHED [build 4/7] RUN mvn -B dependency:go-offline   0.0s
=> CACHED [build 5/7] COPY src ./src                     0.0s
=> CACHED [build 6/7] RUN mvn -B clean package            0.0s
=> [build 7/7] RUN mvn ... copy-dependencies             15.1s   ← only this changed, so only this re-ran

This is why the Dockerfile orders instructions the way it does — COPY pom.xml . and RUN mvn dependency:go-offline happen before COPY src ./src, deliberately. Dependencies (`pom.xml`) change far less often than source code. If we copied everything at once, editing a single Java file would invalidate the dependency-download layer too, and every build would re-download the entire Maven dependency tree. Ordering "things that change rarely" before "things that change often" is the single most important Dockerfile optimization technique.

Multi-stage builds — why our Dockerfile has two FROM lines

Look at the real backend Dockerfile:

FROM maven:3.9-eclipse-temurin-17 AS build   # Stage 1: "build" — has Maven, JDK, full toolchain
WORKDIR /app
...
RUN mvn -B clean package -DskipTests          # produces the .jar

FROM eclipse-temurin:17-jre-jammy             # Stage 2: fresh, slim — just a JRE, no Maven
WORKDIR /app
...
COPY --from=build /app/target/billbull-backend-*.jar app.jar   # only the JAR crosses the bridge

Stage 1 needs the full Maven build toolchain (hundreds of MB) to produce the jar. Stage 2 only needs a JRE (much smaller) to run it. The final image that actually gets shipped and deployed is stage 2 — Maven, the JDK compiler, and all of `~/.m2` never make it into the final image at all. This keeps the deployed image lean and reduces attack surface (no build tools sitting in production).

Real bug we hit: the Playwright classpath problem

This is where multi-stage builds get subtle. We needed the Playwright CLI's install --with-deps chromium command to run during the image build, so Chromium is baked in and every deployed container has it already — no runtime download needed. But we tried three approaches before one worked:

  1. java -cp app.jar com.microsoft.playwright.CLI ... — failed. Why: a Spring Boot "fat jar" (built by spring-boot-maven-plugin) doesn't lay its dependencies out as plain files on a classpath — it nests them inside BOOT-INF/lib/*.jar, in a custom format only Spring Boot's own bootstrap loader knows how to unpack at runtime. A plain java -cp app.jar can only see app.jar's own top-level classes, not what's zipped inside its nested jars.
  2. java -jar playwright-1.49.0.jar ... — failed differently. Why: running a jar with -jar requires that jar's own META-INF/MANIFEST.MF to declare a Main-Class. We checked directly: unzip -p playwright-1.49.0.jar META-INF/MANIFEST.MF — no such line existed. Playwright's own jar is a library, not runnable standalone.
  3. What worked: use Maven's dependency plugin to copy Playwright and its actual runtime dependencies into a plain folder, then run the class directly against that folder as a classpath: 
    RUN mvn -B org.apache.maven.plugins:maven-dependency-plugin:3.6.1:copy-dependencies \
    -DincludeGroupIds=com.microsoft.playwright \
    -DoutputDirectory=/app/playwright-lib \
    -DincludeScope=runtime && ...
RUN java -cp "playwright-lib/*" com.microsoft.playwright.CLI install --with-deps chromium

The lesson generalizes: "run a class from a dependency jar" always needs either (a) that jar's own manifest to declare a runnable main class, or (b) you assemble a real classpath folder with that class's dependencies present, and invoke with -cp folder/* not -jar single-file.jar. We even hit a sub-trap: -DincludeArtifactIds=playwright looked correct but silently only grabbed the one named jar, not its transitive dependencies — we caught this by inspecting the actual build stage's output folder before running the full (slow) build:

docker build --target build -t debug-build-stage .
docker run --rm debug-build-stage ls -la /app/playwright-lib

That command is worth remembering on its own: you can build just one stage of a multi-stage Dockerfile (--target build) and inspect it directly with docker run, without paying for the rest of the build. This turned a ~2 minute guess-and-rebuild cycle into a ~15 second check.

.dockerignore — the file we almost forgot

Just like .gitignore tells Git what to skip, .dockerignore tells docker build what not to send into the build context. Without it, our first frontend build transferred 370MB (all of node_modules) into the build context every single time, even though npm ci inside the container immediately overwrites it. We fixed this after the fact:

# billbull-frontend/.dockerignore
node_modules
dist
dist_check
.vite
.env
.env.local
.env.*.local

Intermediate Part 3 — What is Kubernetes, and why not just Docker?

The problem Kubernetes solves

Docker alone answers "how do I package and run one container." It doesn't answer: What happens when the container crashes — does it restart automatically? What happens when I want zero-downtime updates? How does traffic find the right container out of several? How do I manage TLS certificates for a dozen different client domains without doing it by hand for each one? That's the gap Kubernetes (often shortened "k8s" — k, 8 letters, s) fills: it's an orchestrator that keeps a declared desired state running, automatically, continuously.

If Docker is like knowing how to drive one truck, Kubernetes is a dispatch system for a whole fleet: it decides which truck goes where, replaces a truck automatically if it breaks down, and routes new orders to whichever truck is free — all from a plan you wrote down once, not by manually radioing each driver every time something changes.

The core idea: declarative, not imperative

With systemd (what BillBull uses today), you give imperative instructions: "restart this service now." With Kubernetes, you give a declarative description: "I want 1 replica of this container running, with this much memory, connected to this database." Kubernetes then continuously works to make reality match that description — if the pod dies, Kubernetes notices the actual state (0 running) doesn't match the desired state (1 running) and starts a new one, with no human needing to notice or act.

This is the single biggest conceptual shift from what BillBull runs today:

systemd (today, for nest/hilite/albadar)Kubernetes (now live for max)
You write a .service file, run systemctl startYou write a .yaml manifest, run kubectl apply
Restart=always restarts a crashed process on this one machineKubernetes restarts a crashed pod, and could reschedule it to any healthy node in a multi-node cluster
You manually pick a free port per client (8080, 8081, 8082...)Ingress routes by hostname; every client can use the same port internally
nginx config + certbot, by hand, per clientOne Ingress resource + cert-manager auto-issues and auto-renews certs

What "k3s" specifically is

Kubernetes itself is a specification with a fairly heavy reference implementation. k3s is a lightweight, single-binary distribution of real, conformant Kubernetes — built by Rancher specifically for resource-constrained or simpler environments (edge devices, small clusters, single-node setups like ours on 77.37.49.42). It bundles sensible defaults (Traefik as the Ingress controller, a simple `local-path` storage provisioner) so you don't have to separately install and wire together a dozen components. Everything you learn about "real" Kubernetes applies directly to k3s — the API, the object types, `kubectl` — it's just packaged more conveniently for our use case.

Intermediate Part 4 — The core k8s objects, one at a time

Kubernetes has many object types, but our entire max deployment only needed eight. Understanding each one, in the order they build on each other, covers essentially everything we did.

1. Pod — the smallest deployable unit

A Pod wraps one or more containers that always run together on the same machine, sharing a network address. In practice, 95% of the time (including all of ours) a Pod has exactly one container — the extra wrapping exists for advanced cases (sidecar containers) we didn't need. You almost never create a Pod directly — you create a Deployment, which creates Pods for you.

2. Deployment — "keep N copies of this running"

This is the object that actually says what to run. Our real one:

apiVersion: apps/v1
kind: Deployment
metadata:
  name: max-backend
spec:
  replicas: 1
  strategy:
    type: Recreate
  template:
    spec:
      containers:
        - name: backend
          image: billbull-backend:pilot
          ports: [{ containerPort: 8082 }]

replicas: 1 means "always keep exactly one pod matching this template alive." If it crashes, Kubernetes creates a replacement automatically — this is the direct replacement for systemd's Restart=always, except Kubernetes can additionally reschedule the replacement onto a different machine in a multi-node cluster (not relevant to our single-node setup today, but the mechanism is the same).

strategy: Recreate vs. the k8s default (RollingUpdate): RollingUpdate would start a *new* pod before killing the old one, briefly running both. We explicitly chose Recreate (old pod fully stops, then new one starts) because our storage (ReadWriteOnce PVC) can only be mounted by one pod at a time, and because ddl-auto=update makes running two schema-differing app versions against one DB simultaneously risky.

3. Service — a stable address for a moving target

Pods are ephemeral — they get recreated with new internal IPs constantly. A Service gives a stable name/address that always routes to whichever pod(s) currently match its label selector, load-balancing between them if there's more than one.

apiVersion: v1
kind: Service
metadata:
  name: max-backend
spec:
  selector: { app: max-backend }
  ports: [{ port: 8080, targetPort: 8082 }]

selector: { app: max-backend } is the whole mechanism: any Pod carrying the label app: max-backend (set in the Deployment's template.metadata.labels) automatically becomes a target of this Service. This label-matching, not any explicit list of pod names, is how Kubernetes wiring works everywhere. port: 8080 is this Service's own arbitrary internal number; targetPort: 8082 is where it actually forwards to inside the container — this indirection is exactly why the Ingress (below) never needs to know the real container port.

4. ConfigMap & Secret — external configuration

Both hold key-value data injected into a container as environment variables (via envFrom). The only real difference: Secrets are base64-encoded at rest and treated as more sensitive by tooling (not shown in plain text by default `kubectl describe`). Neither is really "encrypted" in a strong sense by default — the real protection is RBAC (who's allowed to read Secret objects at all) plus, in our case, simply never committing the filled-in file to git.

This is precisely the mechanism that replaced our systemd unit's Environment= lines. SPRING_PROFILES_ACTIVE=client2 in the ConfigMap and SPRING_DATASOURCE_PASSWORD=... in the Secret both become real environment variables inside the container the moment it starts — Spring Boot then reads them exactly the same way it always has.

5. PersistentVolumeClaim (PVC) — durable storage

A container's own filesystem is thrown away the instant it's recreated — anything written there (like an uploaded product image) would vanish on the next restart. A PVC is a request for storage that outlives any individual pod. Our PVC uses k3s's built-in local-path storage class, which just carves out a real directory on the node's disk — conceptually similar to today's local /uploads folder, just managed by k8s instead of directly by the filesystem.

The catch: accessModes: ["ReadWriteOnce"] means only one pod can mount this volume at a time. That's exactly why we pinned replicas: 1 and strategy: Recreate — two pods trying to mount the same ReadWriteOnce volume simultaneously would fail.

6. Ingress — routing by hostname, with automatic TLS

apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  annotations:
    cert-manager.io/cluster-issuer: letsencrypt-prod
spec:
  tls:
    - hosts: ["max.billbull.app"]
      secretName: max-tls
  rules:
    - host: max.billbull.app
      http:
        paths:
          - path: /api
            backend: { service: { name: max-backend, port: { number: 8080 } } }
          - path: /
            backend: { service: { name: billbull-frontend, port: { number: 80 } } }

This is the k8s equivalent of an nginx server { server_name max.billbull.app; ... } block. Traefik (k3s's built-in Ingress controller) reads this object and does the actual routing: requests to max.billbull.app/api/* go to the backend Service, everything else goes to the shared frontend Service. The cert-manager.io/cluster-issuer annotation is what triggers automatic certificate issuance — no manual certbot step.

7. ClusterIssuer + cert-manager — TLS certificates on autopilot

cert-manager is itself a small application that runs inside the cluster (we saw its 3 pods: cert-manager, cert-manager-cainjector, cert-manager-webhook). It watches for Ingress objects with the right annotation, automatically requests a certificate from Let's Encrypt via the ACME protocol, proves domain ownership (HTTP-01 challenge — briefly serving a special response at a well-known URL that Let's Encrypt checks), and stores the resulting certificate in a Secret (max-tls) that the Ingress then uses. It also auto-renews before expiry. This replaced every manual certbot renew the old guide mentioned.

8. Namespace — a folder for related objects

Just a logical grouping. All BillBull clients share one namespace, billbull — matching today's model of one server hosting multiple clients side by side. Namespaces matter more when you want hard isolation (RBAC boundaries, resource quotas per team) — not needed here yet.

Intermediate Part 5 — BillBull's actual architecture, mapped to k8s

Before: two servers, systemd, nginx

Every client backend was its own JVM process, its own systemd unit, its own port, with DB credentials injected via Environment= lines or (as we discovered for max) sometimes just hardcoded in the checked-in application-<profile>.properties file with no override at all.

After (so far): 77.37.49.42 is a single-node k3s cluster

One node (still the same physical server) runs the k3s control plane and all workloads. max is fully migrated; nest, hilite, albadar are currently stopped (a deliberate choice made to free ports 80/443 for Traefik during this pilot — see Part 6). Postgres was not containerized — it stays exactly where it was, running directly on the host via its own systemd/apt-installed service, and pods reach it over the network by IP.

Why Postgres stayed outside the cluster

This is a deliberate, common pattern called "day-2 stateful services stay external." Running a production database *inside* Kubernetes is possible but adds real operational complexity (StatefulSets, storage replication, backup tooling built for k8s specifically) that isn't worth taking on here — the existing Postgres install already works, has existing backups, existing tooling. The only thing that had to change was making it reachable from the new pod network:

ChangeWhy
listen_addresses = '*' (was localhost-only, commented out)Postgres was refusing any connection not from the same machine's loopback — a pod, despite being on the same physical machine, has its own network namespace and connects via a real (if internal) IP, not 127.0.0.1

This mirrors exactly the concern raised earlier in this project's planning: "where do uploaded images live in k8s, and is Postgres containerized?" The answer that emerged from actually doing it: Postgres stays external (confirmed the right call), and file uploads use a PVC per client for now — with a note that this constrains every client backend to replicas: 1 forever unless uploads move to object storage (S3/MinIO) later.

Advanced Part 6 — The real migration, narrated end-to-end

This is the actual sequence of what happened on 77.37.49.42, in order, with the reasoning behind each decision — not the idealized plan, the real one, including the parts where the plan changed mid-execution.

Step 1 — Reconcile git state before touching anything

Before any infrastructure work, git pull origin main on the server failed: local uncommitted edits to application.properties conflicted with incoming changes. Investigation showed the local edit (stripping server.port and dev-only DB defaults from the base properties file) was a deliberate, safe hardening someone had made directly on the server — but had never been committed, so it kept re-conflicting on every pull. Lesson: infrastructure changes should never proceed on top of unexamined uncommitted state — git status/git diff first, always. The fix was committed properly to the repo so it stops recurring.

Step 2 — Build the Docker images (see Part 2 for the Playwright saga)

Built directly on the target server rather than pushing to a registry, since this was a pilot — a deliberate scope-reduction (see Part 8 for what changes once this goes to multiple clients).

Step 3 — Install k3s, then immediately discover the port conflict

The instant curl -sfL https://get.k3s.io | sh - completes, k3s starts Traefik, which by default wants to bind host ports 80 and 443 — the same ports nginx already owned for nest/hilite/albadar. We verified this precisely rather than guessing:

kubectl get svc -n kube-system traefik -o wide
# NAME      TYPE           EXTERNAL-IP    PORT(S)
# traefik   LoadBalancer   77.37.49.42    80:32563/TCP,443:32322/TCP

Traefik's Service already listed the host's real IP as its EXTERNAL-IP — meaning k3s's built-in load-balancer implementation (called ServiceLB, sometimes "klipper-lb") intended to bind the real host ports. It hadn't succeeded yet only because nginx got there first at boot. This was a live, so-far-invisible race condition that could have flipped and taken down three production clients with no warning, the next time either service restarted.

The original plan (written before touching the server) was to run Traefik on alternate ports so nginx could keep serving the other three clients during testing. The plan changed at execution time: given `nest`/`hilite`/`albadar` weren't needed at that moment, the deliberate decision was made to stop nginx outright and let Traefik take 80/443 immediately — trading "keep everything running" for "move faster on the pilot," a real tradeoff, made consciously, not by accident.

sudo systemctl stop nginx
sudo systemctl disable nginx

Step 4 — Diagnosing "is Traefik actually working" the right way

Real bug (well, false alarm) we hit: `ss` showed nothing listening

After stopping nginx, sudo ss -tlnp | grep -E ':80|:443' returned nothing — no process at all appeared to be listening. This looked alarming (did Traefik fail to start?), but curl -sI http://127.0.0.1:80 immediately returned a real HTTP response (Traefik's own 404 page, since no Ingress existed yet). The lesson: k3s's ServiceLB does port-forwarding via iptables/nftables DNAT rules from inside its own pod's network namespace, not a classic host-level bind()+listen() — so ss, which only sees host-level sockets, is the wrong tool to check whether it's working. curl against the actual port is the correct verification. This is a good general principle: when a "is it listening" check disagrees with an "is it actually answering" check, trust the one that tests real behavior.

Step 5 — Postgres network access (see Part 5 for the why; here's the how)

Real bug we hit: a `sed` that silently did nothing

The first attempt to uncomment listen_addresses used sed -i "s/^listen_addresses = 'localhost'/listen_addresses = '*'/" — anchored on a line starting exactly with listen_addresses. It ran without error and changed nothing, because the real line in the file was #listen_addresses = 'localhost' — commented out, so it didn't start with listen_addresses at all, it started with #. sed doesn't warn you when its pattern matches zero lines; it just succeeds silently having done nothing. The lesson: always grep -n the exact current text before writing a substitution command — never assume the format of a config line from memory or from documentation, verify it directly on the real file.

Once the exact line was confirmed (line 60, with a trailing comment), a line-number-anchored sed fixed it correctly, and this was verified two ways — checking the config value itself (SHOW listen_addresses) and checking the actual listening socket (ss -tlnp | grep 5432 now showing 0.0.0.0:5432).

Step 6 — Secrets, filled in on the server, never in git

The Secret template (k8s/max/secret.template.yaml) was copied to secret.yaml, filled with the real DB URL/user/password and JWT secret directly on the server via nano, applied with kubectl apply, and confirmed with git status --short k8s/max/ to show nothing pending — because k8s/**/secret.yaml is in .gitignore. This is a deliberate two-file pattern worth internalizing: a template with placeholder values is safe to commit and documents the shape of what's needed; the real, filled-in file never touches version control.

During this project, an earlier draft of RUNBOOK.md briefly had the real DB password written directly into the markdown documentation (not the Secret file) as an example — this was caught by an automated safety check before it reached the public GitHub repo, and fixed by keeping the doc's guidance structural ("the systemd unit has no password override, so the live value is whatever's in the properties file — verify it directly, don't paste it here") rather than stating the literal secret value. The general rule: documentation should explain how to find a secret, never contain the secret itself.

Step 7 — Deploy, verify, cut over

Applying the Deployment produced a pod that went straight to Running, and its logs showed live Hibernate SQL queries executing — strong evidence the DB connection worked on the first real attempt (a payoff from carefully verifying each prerequisite — Postgres listening, `pg_hba.conf`, Secret values — independently before this step, rather than debugging all of it at once from inside a failing pod). The frontend, Ingress, and TLS certificate all came up cleanly. Because DNS for max.billbull.app already pointed at 77.37.49.42 (it had been serving this client from this exact IP for a long time already), no DNS change was needed at all — the Let's Encrypt HTTP-01 challenge resolved immediately.

Final verification was manual, in a real browser, against the real domain: login, existing data visibility, PDF generation, and file upload were each checked by hand — because passing type-checks or a clean `kubectl get pods` proves the infrastructure is technically running, not that the actual product works end-to-end for a real user.

Advanced Part 8 — Day-2 operations: how to actually run this

Deploying a code change

Nothing auto-rebuilds or auto-redeploys — this is the same manual trigger model as systemd's restart, just via different commands:

# backend
cd ~/Billbull/Billbull_Retail && git pull origin main
cd billbull-backend
docker build -t billbull-backend:pilot .
docker save billbull-backend:pilot | sudo k3s ctr images import -
kubectl rollout restart deployment/max-backend -n billbull
kubectl rollout status deployment/max-backend -n billbull   # watch it succeed

# frontend — remember, this is SHARED across every client on this server
cd ../billbull-frontend
docker build -t billbull-frontend:pilot .
docker save billbull-frontend:pilot | sudo k3s ctr images import -
kubectl rollout restart deployment/billbull-frontend -n billbull

kubectl rollout restart is necessary (not optional) even though the image tag (:pilot) didn't change — Kubernetes doesn't know the content behind that tag changed unless told to restart explicitly. Config changes (editing a ConfigMap/Secret) similarly need an explicit rollout restart afterward — env vars are only read at container start, not live-reloaded. Editing the Deployment manifest itself (e.g. changing memory limits) and re-applying it *does* trigger a rollout automatically, since the pod spec itself changed.

What's still manual vs. what k8s automates, honestly

  • Automated now: crash recovery (pod restarts itself), TLS issuance/renewal, routing by hostname.
  • Still manual: building and importing new images (no CI/CD pipeline yet — see below), the initial per-client Secret/PVC/DB-access setup for each new client migration.
  • Deliberately deferred, not solved: uploads still constrain every client to replicas: 1 — genuine horizontal scaling or true zero-downtime rolling updates need uploads moved to object storage (S3/MinIO) first.

What changes once more clients get migrated

Building images by hand on the server and k3s ctr images import-ing them was a deliberate pilot shortcut — fine for one client, but it doesn't scale to rebuilding on every server for every client. The next real step (already flagged in the main migration guide) is standing up a proper registry (GitHub Container Registry) so one build produces one image that every server/client pulls from, plus templating the per-client manifests via Helm instead of hand-copying k8s/max/ for each new client.

The one-paragraph mental model, if you remember nothing else

You write YAML files describing what you want running (a Deployment), how it should be reachable internally (a Service) and externally (an Ingress), and what config/secrets/storage it needs (ConfigMap, Secret, PVC). You hand all of that to Kubernetes via kubectl apply, and Kubernetes continuously works to keep reality matching that description — restarting crashed pods, renewing certificates, routing traffic — without you having to notice and react by hand, the way systemd + nginx + certbot required.
