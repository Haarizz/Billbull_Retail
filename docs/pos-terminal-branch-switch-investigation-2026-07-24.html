Investigation Report — BillBull POS

POS Terminal & Session Restoration — Branch-Switch Investigation

Repo: BillBull Retail Branch: feature/billpos Date: 2026-07-24
Verdict: architectural bug, not intentional branch-isolation behavior — confirmed by direct code read, not inference.

Executive summary

Terminal identity is keyed on a single global device_fingerprint value with a database-level unique constraint, but a physical till is expected to hold one terminal identity per branch. There is no composite (fingerprint, branch) identity anywhere in the stack. Every time the registration code meets a fingerprint that already belongs to a different branch, it detaches the fingerprint from the old terminal row and mints a new one for the current branch — a one-way, destructive hand-off.

Branch switching on the frontend never signals the POS page at all, so the stale terminal/session state simply sits there until the page remounts, at which point the hand-off above fires.

Nothing here is a security or isolation feature. It is a missing key column and a missing event listener.

Root cause, in order of appearance

1. The backend match key excludes branch by design, not by accident

PosTerminalService.registerOrRefresh() — billbull-backend/.../pos/terminal/PosTerminalService.java:50

Two lookups are tried, in order:

StepLookupBranch-aware?
1repo.findByTerminalId(terminalId)Only used after the row is found — if branch doesn't match, falls through silently (no error, no cleanup)
2repo.findByDeviceFingerprint(deviceFingerprint)No. Fingerprint is the sole key — PosTerminalRepository.java:12

The fingerprint column carries a database-level unique index (PosTerminal.java:10), so it is structurally impossible for the same device to hold two live terminal rows — one per branch — at once. When step 2 finds the fingerprint parked on a different branch's terminal:

// PosTerminalService.java:107-111
} else {
    // Device fingerprint belongs to another branch. Unlink it so the new branch can claim it.
    t.setDeviceFingerprint(null);
    repo.saveAndFlush(t);
}

The old terminal is stripped of its fingerprint — permanently unreachable by any future lookup — and a brand-new terminal row is created for the current branch (PosTerminalService.java:114-153). The comment even documents the hand-off as deliberate — just scoped wrong. This directly consumes one terminal slot per branch per device, forever, since the orphaned rows are never deleted (only eventually caught by the unrelated auto-archive sweep on lastActivityAt).

2. Frontend identity keys are global, not per-branch

POSSales.jsx:1826-1837:

let fp = localStorage.getItem('billbull:pos:device_fingerprint');   // one per browser, forever
...
const cachedTerminalId = localStorage.getItem('billbull:pos:terminal_id') || null; // last branch registered wins

Both keys are global — there is no _${branchId} suffix. Contrast with the scanner-config key three lines away in the same file, which is branch/terminal-scoped (billbull:pos:scanner:${branchId}:${terminalId}) — the pattern for doing this correctly already exists in the codebase, it just wasn't applied to terminal identity itself.

Net effect: after registering on Branch B, localStorage's terminal_id now points at Branch B's terminal. Any later attempt to resolve Branch A's terminal sends the wrong ID.

3. Branch switching never notifies the POS page

BranchContext.jsx:284-300 switchBranch() swaps the branch-scoped JWT and fires:

window.dispatchEvent(new CustomEvent('billbull:branch-changed', { detail: { branchId: nextValue } }));

17 other pages (Sales Orders, Sales Invoice, LPO, GRN, Ledger, Journal Voucher, etc.) listen for this event and re-fetch. POSSales.jsx has no listener for it at all. The terminal-registration effect is also mounted once, empty dependency array:

useEffect(() => { ... }, []);   // POSSales.jsx:1860 — never re-runs on branch change

So switching branch while POS is mounted changes nothing about currentTerminal/currentSession — the stale state is only disturbed on the next full remount of the page, which is exactly when the destructive hand-off in §1 fires.

4. Session lookup silently returns "nothing" instead of surfacing the real cause

GET /api/pos/sessions/active?terminalId=... → PosSessionService.getActiveSession() (PosSessionService.java:221-237) resolves branch from the JWT server-side and queries:

repo.findByBranchIdAndTerminalIdAndStatus(branchId, terminalId, PosSessionStatus.OPEN)

This composite lookup is correctly branch-aware. But if the client sends a terminalId from localStorage that belongs to a different branch than the current JWT, the row simply doesn't exist under that combination — the endpoint returns 204 No Content. There's no distinct signal for "your terminal moved branches"; it looks identical to "you never had a session," which is why the session appears to just vanish.

Each round trip through a branch burns one terminal slot on that branch and abandons the previous terminal row permanently. This is precisely both symptoms reported: terminal-slot exhaustion (issue 1) and lost session/terminal association on return to a branch (issue 2).

Is any of this intentional isolation?

No. Branch scoping is enforced correctly in the one place it matters most for security — PosSessionService resolves the branch from the server-side JWT, not from client input, so a session can never be read cross-branch. That's the right pattern. The bug is one layer up: terminal identity has no branch dimension at its storage key, so the system cannot represent "this device has a distinct, simultaneously-valid terminal in both Branch A and Branch B" — which is exactly what a till that's shared or reassigned across branches needs.

Files responsible

Backend pos/terminal/PosTerminalService.java:50-154 — registerOrRefresh, the actual bug

Backend pos/terminal/PosTerminalRepository.java:12-13 — missing composite finder

Backend pos/terminal/PosTerminal.java:10 — unique index scoped to fingerprint alone

Backend pos/session/PosSessionService.java:221-237 — correct branch-aware lookup (no change needed), but silent 204 on mismatch is a diagnosability gap

Frontend pages/Sales/POSSales.jsx:1824-1860 — global (non-branch-scoped) localStorage keys, mount-only effect

Frontend context/BranchContext.jsx:284-300 — emits billbull:branch-changed, which POSSales never listens for

Frontend/backend split: both sides need changes — this is not fixable from either side alone.

Recommended fix (preserves current architecture)

  1. Backend — make the identity key composite. Add a unique index on (device_fingerprint, branch_id) instead of device_fingerprint alone, and add findByDeviceFingerprintAndBranchId(fp, branchId) to the repository. Use it as the primary lookup in registerOrRefresh before falling back to a global fingerprint search. This alone lets the same device hold one live terminal row per branch simultaneously — no more forced hand-off.
  2. Backend — stop destructively unlinking on cross-branch match. Once lookup is composite, a cross-branch fingerprint hit means "this device has a different terminal on that other branch" — not an error, just create/reuse the branch-specific row and leave the other branch's row untouched.
  3. Frontend — scope the localStorage keys per branch, following the existing scanner-config pattern already in the file: billbull:pos:terminal_id:${branchId} and (optionally) keep device_fingerprint global since the fingerprint should describe the physical device, not the branch — branch scoping belongs on the terminal-ID cache, not the fingerprint.
  4. Frontend — react to billbull:branch-changed in POSSales.jsx. Re-run the terminal-registration/session-restoration effect (extract it out of the mount-only useEffect([]) into a callback triggered both on mount and on the branch-changed event) so a live branch switch immediately looks up the correct branch's terminal instead of waiting for a remount to trigger the corrupting hand-off.
  5. Backend — surface the mismatch instead of a bare 204. When getActiveSession finds a terminal for a different branch than the caller's, return a distinguishable response (e.g. 409 with a "terminal registered to another branch" body) so the frontend can react explicitly instead of silently treating it as "no session."
  6. One-time cleanup migration to merge/close out currently-orphaned terminal rows (fingerprint stripped, branch mismatch history) so terminal-slot counts return to accurate before the fix ships.

None of this requires touching the auto-archive lifecycle, RBAC, or session-locking logic — those are already correct and branch-aware.
