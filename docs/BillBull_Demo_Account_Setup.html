Overview & Account Details

This guide deploys a dedicated, isolated demo environment so prospects and customers can experience BillBull Retail Enterprise OS's full application flow, UI, and feature set — without any risk to live client data. The demo runs on its own Postgres database (billbull_demo) and its own backend instance (port 8083), following the exact same per-client isolation pattern already used for royaltools, leroyalflowers, and leroyalgifts.

Systemd Service

Profile demo, port 8083 — matches application-demo.properties already in the repo.

BASH
cat > /etc/systemd/system/billbull-demo.service << 'EOF'
[Unit]
Description=BillBull - Demo
After=network.target postgresql.service

[Service]
User=root
WorkingDirectory=/root/billbull/billbull-backend
Environment=ADMIN_INITIAL_PASSWORD=<set-a-strong-temp-password-here>
ExecStart=/usr/bin/java -jar target/billbull-backend-0.0.1-SNAPSHOT.jar --spring.profiles.active=demo
Restart=always
RestartSec=10
StandardOutput=journal
StandardError=journal

[Install]
WantedBy=multi-user.target
EOF

systemctl daemon-reload
systemctl enable billbull-demo
systemctl start billbull-demo
systemctl status billbull-demo
Why set ADMIN_INITIAL_PASSWORD: on first boot, RBACInitializer auto-creates a default admin user (role ADMIN) plus a default branch ("Main Branch" / BR-01) if none exist. Without this env var it generates a random password and only prints it to the journal — set it explicitly so you control it from the start.

Nginx Server Block

Append to the existing /etc/nginx/sites-available/billbull (same pattern as the other 3 clients), and add demo.billbull.app to the listen 80 redirect block's server_name.

NGINX
server {
    listen 443 ssl;
    server_name demo.billbull.app;
    root /var/www/billbull;
    index index.html;
    client_max_body_size 300M;

    ssl_certificate /etc/letsencrypt/live/demo.billbull.app/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/demo.billbull.app/privkey.pem;
    include /etc/letsencrypt/options-ssl-nginx.conf;
    ssl_dhparam /etc/letsencrypt/ssl-dhparams.pem;

    location / { try_files $uri $uri/ /index.html; }
    location /assets/ { expires 1y; add_header Cache-Control "public, immutable"; }
    location /uploads/ { proxy_pass http://127.0.0.1:8083/uploads/; }
    location /api/ {
        proxy_pass http://127.0.0.1:8083;
        proxy_http_version 1.1;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_set_header Authorization $http_authorization;
    }
}

Making It Resettable

Take a "golden" snapshot after the in-app configuration and sample data are in place, so resets restore the fully-configured demo state rather than the bare seed.

BASH
mkdir -p /root/billbull/backups
pg_dump -U postgres billbull_demo > /root/billbull/backups/demo_golden.sql

Reset Script

BASH
cat > /root/billbull/scripts/reset_demo.sh << 'EOF'
#!/bin/bash
set -e
systemctl stop billbull-demo
sudo -u postgres dropdb billbull_demo
sudo -u postgres createdb billbull_demo -O postgres
sudo -u postgres psql -d billbull_demo -f /root/billbull/backups/demo_golden.sql
systemctl start billbull-demo
echo "Demo reset complete: $(date)"
EOF
chmod +x /root/billbull/scripts/reset_demo.sh

Run it manually before each demo, or schedule it (e.g. nightly at 3am Gulf time):

CRON
0 3 * * * /root/billbull/scripts/reset_demo.sh >> /var/log/billbull-demo-reset.log 2>&1
Isolation guarantee: this app is multi-tenant by separate database (same pattern as royaltools / leroyalflowers / leroyalgifts). billbull_demo is fully isolated — there is no code path by which demo activity could touch real client data.
