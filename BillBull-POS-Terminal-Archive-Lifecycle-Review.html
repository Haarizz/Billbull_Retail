POS Terminal Archive Lifecycle Review

The premise behind this review — that Archive permanently frees the device and forces a new registration — turns out not to match the code. The backend already supports exactly the reversible behavior the concern describes; the actual gap is that this only exists on the admin side, and the cashier-facing dialog can't tell an admin what to do because it doesn't know why the terminal is unavailable.

Part 01

What already exists — read before anything else

Before recommending an architecture, it's worth being precise about what's already shipped, because it changes the shape of the problem. The concern as stated assumes ARCHIVED is a one-way door. It isn't — not on the backend, and not even in one place on the frontend. It's a one-way door in exactly one screen: the cashier-facing POS terminal-pairing dialog.

Reframing the question

The concern asks "should ARCHIVED be Option A (temporary, restorable) or Option B (permanent, forces re-registration)?" The backend already implemented Option A in full, correctly, months ago. The actual open question is narrower and different: the system has no Option B at all — no status currently means "this device/terminal relationship is permanently over, register fresh." DECOMMISSIONED is the obvious candidate for that role, and it's sitting there unused. The rest of this review is built around that reframing.

Part 02

Note on MAINTENANCE reconnect gating

Registration correctly blocks MAINTENANCE (PosTerminalService.java:67-72, 94-99), but the heartbeat guard only lists ARCHIVED/DECOMMISSIONED/BLOCKED, not MAINTENANCE (PosTerminalService.java:169-173). A terminal already holding a valid session that gets put into MAINTENANCE mid-shift would keep heartbeating successfully until that session naturally ends, rather than being cut off immediately. This may be intentional (let an active sale finish before maintenance takes effect) or may be an oversight — flagged as a decision point in Part 09, not assumed either way.

Part 03

Resolving ARCHIVED: Option A, confirmed correct — and already built

Directly answering the task's central question: ARCHIVED should be, and already is, Option A — a temporary inactive state, restorable, with the device-terminal relationship preserved and no new registration required. Option B's behavior (permanent, forces new registration, consumes a new slot) is a real and needed concept, but it belongs to a different status — DECOMMISSIONED — not to ARCHIVED.

PropertyOption A (ARCHIVED, as built)Option B (belongs to DECOMMISSIONED instead)
ReversibleYes — admin Restore, already liveNo, by design
Device-terminal relationshipPreserved — same PosTerminal row, same deviceFingerprintEnded — device must pair to a brand-new row
New registration requiredNoYes
Consumes a new slotNo — restoring reuses the existing (already-excluded) rowYes — correctly, since it's genuinely a new terminal identity
Typical triggerAuto-archive after inactivity (PosTerminalAutoArchiveJob), or a deliberate "temporarily take this till out of service" admin actionTerminal hardware retired/replaced, branch closed, device permanently reassigned, admin explicitly ends the relationship

Why this split is the right one, not just the existing one: the whole point of the auto-archive sweep (PosTerminalAutoArchiveJob, daily at 02:00, per-branch opt-in, 30-day default threshold) is to quietly reclaim slots from terminals nobody is actively touching — a seasonal till, a device that broke and was forgotten, a till from a temporarily-closed department. None of those situations are "this hardware is gone forever" — they're "nobody's used this in a month, and if they come back, restoring should be one click, not a full re-registration that eats another slot and loses the device's history/fingerprint continuity." That is precisely Option A, and building the auto-archive job to feed into anything else would have made the whole feature actively hostile to the exact scenario it was built to handle.

Part 04

ARCHIVE and DECOMMISSION should be separate concepts — and already are, on paper

Yes, they should be separate — and the enum already reflects that design intent (DECOMMISSIONED is deliberately excluded from the same queries as ARCHIVED everywhere, not merged with it). The gap isn't conceptual; it's that DECOMMISSIONED was defined and wired into every read-side check but never wired into a write-side trigger. It's a state nothing can ever enter.

ARCHIVEDECOMMISSION
Semantic"Not in use right now""Retired — this pairing is finished"
Who triggers itAuto-archive job (system) or admin "Archive Now"Admin only — a decommission should never happen automatically, since it's irreversible and slot-consuming to undo
ReversibleYesNo
What happens to the deviceTold terminal is unavailable; if it's their terminal, an admin can bring it back exactly as it wasTold to register as a new terminal — correctly, since there's nothing left to restore

This mirrors a pattern already established elsewhere in the POS device layer: cash drawers, scanners, printers, and hardware profiles all have their own DECOMMISSIONED-equivalent status that is actively assigned somewhere in their respective services (PosCashDrawerService.java:66, PosScannerService.java:56, PosDeviceController.java:133, HardwareProfileService.java:82, PosPrinterService.java:86). Terminals are the one device type in this family where that final step was left unfinished. This isn't a new concept being introduced — it's completing a pattern the rest of the module already follows.

No circular or ambiguous paths: ARCHIVED and DECOMMISSIONED are both reachable only via explicit admin action or the auto-archive job (for ARCHIVED only — DECOMMISSIONED is never automatic, per Part 04). BLOCKED and MAINTENANCE are orthogonal holds that suspend from, and return to, whatever operational status the terminal was in before — they don't interact with the ARCHIVE/DECOMMISSION axis at all.

Part 06

Terminal licensing behavior

Directly answering the task's licensing question: archiving should free a slot immediately; only decommissioning should also permanently forfeit the ability to silently reclaim it. This is already exactly how ARCHIVED behaves today. The only new piece is confirming DECOMMISSIONED's behavior once it's wired up.

StatusCounts against maxTerminalsPerBranch?Rationale
ACTIVE / IDLE / OFFLINE / STALE / MAINTENANCE / BLOCKED / INACTIVEYesAll represent a live, reusable terminal identity — even BLOCKED/MAINTENANCE, since those are meant to be temporary and the slot should stay reserved for that specific terminal, not freed for someone else to grab
ARCHIVEDNo — already correct todayThe whole point of auto-archive is to free up slots occupied by dormant terminals without losing the ability to bring them back exactly as they were
DECOMMISSIONEDNo (once wired — already excluded from the same query as ARCHIVED)The relationship is over; the freed slot is available for a genuinely new terminal registration, which is the correct incentive since decommissioning is an admin-only, deliberate, irreversible action

One nuance worth deciding explicitly rather than assuming: today, BLOCKED and MAINTENANCE terminals still consume a slot indefinitely. That's defensible (they're not gone, just paused, and a malicious or careless admin shouldn't be able to free up "extra" slots just by blocking terminals) — but it means a branch that has, say, 3 of its 5 terminals permanently BLOCKED for a legitimate reason (recurring fraud, retired-but-not-yet-decommissioned hardware) is effectively down to 2 usable slots with no path to reclaim them short of unblocking or decommissioning. Recommend surfacing this explicitly in the admin UI (a "blocked/maintenance terminals are still counted against your limit" note) rather than changing the underlying behavior — the current behavior is correct, it just isn't visible.

Part 09

Should the current implementation change?

Yes, but narrowly. The lifecycle architecture itself does not need to change — Part 03 confirms ARCHIVED's existing design (Option A) is the correct one. What needs to change is completing one missing state transition and fixing the frontend's blindness to the states the backend already distinguishes.

AreaChange needed?Why
ARCHIVED semanticsNo changeAlready correctly Option A — reversible, no forced re-registration, slot already freed
DECOMMISSIONED transitionsChange neededCurrently unreachable — needs an admin-triggered "Decommission" action added to close the gap the task is actually describing
Device-side dialogChange neededCurrently undifferentiated across 4 distinct causes; the backend already returns enough information to fix this without any new backend work
Slot-count visibility for BLOCKED/MAINTENANCEMinor UI addition recommendedBehavior is correct, but currently invisible to admins, which risks confusing "why can't I add a terminal" support tickets
MAINTENANCE heartbeat gating inconsistency (Part 02 note)Needs an explicit decision, not necessarily a code changeCould be intentional (let active session finish) — confirm intent before treating as a bug
INACTIVE as restore's landing stateWorth a naming/behavior reviewRestoring an ARCHIVED terminal lands it in INACTIVE rather than directly in ACTIVE/IDLE — functionally fine (next heartbeat resolves it) but worth confirming this is deliberate rather than an artifact, since INACTIVE isn't otherwise a state anything else in the lifecycle produces

None of these are architecture changes — they're completing and exposing an architecture that was already built correctly in its core design. There is no case here for revisiting whether ARCHIVED should behave differently than it does today.

Part 10

Implementation plan — preserving the existing architecture

Four small, independently deployable changes. None require a new migration for the core concept (the DECOMMISSIONED enum value and all its guard-clause checks already exist) — only new write-side code and frontend logic.

1. Wire up the Decommission action (backend)

  • New endpoint, mirroring the existing archive/restore pair: POST /api/pos/terminals/{id}/decommission (with an optional reason, matching archiveReason's pattern).
  • New service method in PosTerminalLifecycleService/PosTerminalService, following the exact structure of the existing archive() method — sets status = DECOMMISSIONED, no restore path offered (intentionally no PosTerminalStatus.DECOMMISSIONED → ACTIVE transition should ever be added to restore()).
  • Recommend gating this behind a confirmation step in the API/UI ("this cannot be undone") rather than a new backend safeguard, since the existing pattern (archive/restore) doesn't have extra backend confirmation machinery either — consistency with the established pattern.
  • Optional: a retention-policy note (not a scheduled job requirement for v1) that ARCHIVED terminals older than N days become eligible for decommission review in the admin UI — surfaced as a filter/sort, not an automatic transition, keeping DECOMMISSIONED admin-only per Part 04.

2. Admin UI: Decommission button + Decommissioned view

  • POSTerminals.jsx: add a "Decommission" action alongside the existing Archive/Restore actions, with a confirmation dialog stating the irreversibility explicitly.
  • New filter/tab for Decommissioned terminals, separate from Archived (Part 08).
  • Slot-usage indicator near Add Terminal (Part 06/08).

3. Device-side dialog: parse status, branch UI (frontend only)

  • In POSSales.jsx's 403 handler (~line 1705), instead of storing the raw message string only, parse which of ARCHIVED/BLOCKED/MAINTENANCE/DECOMMISSIONED it corresponds to (simple string match against the known backend message format, or — cleaner — have the backend return a structured error code alongside the message, which is a small, low-risk backend addition to the existing exception response rather than a new endpoint).
  • Render the dialog (~line 7906) conditionally per Part 08's table — heading, body, and action button all keyed off the parsed status.
  • No backend contract change is strictly required (string-matching the existing message works), but adding a structured field (e.g. errorCode: "TERMINAL_ARCHIVED") to the existing ResponseStatusException body is the more robust option and touches only the exception-throwing lines already identified (PosTerminalService.java:67-72, 94-99, 169-173) — no new migration, no new endpoint.

4. Decide and document the two open questions from Part 09

  • Confirm whether MAINTENANCE should block an in-progress heartbeat immediately or allow the current session to finish — update PosTerminalService.java:169-173 only if the decision is "block immediately."
  • Confirm INACTIVE-as-restore-landing-state is intentional; if not, change PosTerminalService.java:252 to set ACTIVE/IDLE directly based on the same recent-heartbeat check keepActive() already uses (PosTerminalLifecycleService.java:188-200) rather than introducing a code change purely to standardize naming.
Why this plan preserves the existing architecture

Every change above extends the existing archive/restore pattern by one more sibling action (decommission) and makes the frontend aware of distinctions the backend has made since the auto-archive feature shipped. Nothing here revisits the STALE/OFFLINE/auto-archive machinery, the slot-counting logic, or the ARCHIVED semantics — all of which this review confirms are already correctly designed. The scope is deliberately the smallest change that closes the actual gap: a permanent-retirement path that was designed into the enum but never finished, and a UI that doesn't yet use the information the backend already provides.
