Part 02

Dependency graph — correct implementation order

Nine layers, strictly sequential where marked, parallelizable where marked. No circular dependencies: each layer only depends on layers above it.

Layer 1 — Tenant Storage Configuration
Decide + provision tenant.storage-key for all 14 tenants. Pure config/admin, zero code.
↓
Layer 2 — Storage Abstraction Core
StorageProvider interface, StorageKey value type, LocalStorageProvider, TenantFileStorageService. No existing call site touched yet.
↓
Layer 3 — Metadata Table
stored_file entity + repository + Flyway V43. Wired into TenantFileStorageService.store() so it's populated from Layer 4 onward. Independent of Layer 4/5 — can build in parallel with them.
↓
Layer 4 — Static Serving + Backward-Compat Route
StaticResourceConfig fallback handler. Additive, safe to ship standalone ahead of everything below — this is deliberately out of strict sequence because it has no dependency on Layers 2-3 and de-risks Layer 6 by existing before any DB row changes.
↓
Layer 5 — Call-Site Refactor (9 sites)
FileUploadUtil, EmployeeServiceImpl, ProductImageStorageService (2 entry points), BrandLogoStorageService, CompanyProfileService, BranchService, SalesOrderController (extract to service), ReceiptVoucherService, TaxService. Each depends only on Layer 2 — parallelizable across developers, sequential only in the sense that Layer 2 must be done first.
↓
Layer 6 — DTO/Response Mapping (Option C resolution)
4 new thin wrappers (CompanyProfile, Employee, SalesOrderAttachment, ReceiptVoucher) + 5 existing-mapper edits. Depends on Layer 2 (needs resolvePublicUrl() to exist) and logically follows Layer 5 (no point resolving URLs for paths not yet being written in the new format) but does not depend on Layer 5 being complete for every module — can start on already-refactored modules while Layer 5 continues on others.
↓
Layer 7 — Database Migration (10 columns → logical key)
Requires Layers 2, 4, 5, 6 all complete first — this is the one truly hard sequencing constraint: rewriting DB values before the code that resolves them exists would break every image/logo/attachment in production.
↓
Layer 8 — Storage Migration (physical file copy, per tenant)
Can run in parallel with Layer 7's DB rewrite for the same tenant (they operate on different systems — filesystem vs. database) but both must complete and both must pass validation before that tenant's Layer 9 cutover.
↓
Layer 9 — Validation, Testing, Cutover
Per-tenant rollout, one at a time, each independently deployable and independently rollback-able.
Why Layer 4 is deliberately out of strict sequence

The blueprint's migration plan (Part 12) needs the backward-compatible fallback route to exist before any DB row is rewritten, but the fallback route itself has zero code dependency on the storage abstraction (Layers 2-3) — it's a second static resource handler. Shipping it early, standalone, removes it from the critical path of the harder refactor work and gives the team a trivially low-risk first deploy to validate the deployment pipeline itself before touching anything riskier.

Part 04

Database migration strategy

Two distinct migrations, deliberately separated: (A) the additive stored_file table, and (B) the rewrite of 10 existing path/URL columns to the logical-key format. A is zero-risk; B is the one migration in this whole roadmap that touches live production data and needs the full discipline below.

Migration A — stored_file table (Flyway V43)

Purely additive — CREATE TABLE IF NOT EXISTS-guarded per the project's existing Flyway convention (confirmed this session: idempotent, to_regclass-style guards are the house style). No existing row touched. Safe to run against every tenant DB immediately, independent of everything else in this roadmap. Zero downtime, zero data-loss risk.

Migration B — path/URL column rewrite (10 columns, 9 entities)

This is a data migration, not a schema migration — no Flyway ALTER needed (column types don't change, only values), but it must run per-tenant, guarded, and only after that tenant's Layer 5/6 code is deployed (per the dependency graph, Part 02).

PropertyDesign
IdempotencyEach UPDATE is a string-prefix transform (strip old format, apply new). Guard every statement with a WHERE column LIKE 'old-pattern%' AND column NOT LIKE 'new-pattern%' so re-running the script after partial success only touches unconverted rows — safe to re-run to completion after any interruption.
Existing data migrationPer tenant DB (already isolated 1:1), one script touching all 10 columns in a single transaction — either the whole tenant's rewrite commits or none of it does, avoiding a half-migrated intermediate state for that tenant.
Backward compatibilityNot needed at the DB layer itself if Layer 4 (static-resource fallback route) is live first — old cached URLs still resolve physically even after the DB stops producing them, because the fallback route serves the old flat-path files independent of what the DB currently says.
ValidationPre-migration: snapshot row counts and a content hash of a sample per column. Post-migration: re-run and diff — same row counts, and an HTTP-fetch spot-check (reusing the checklist already defined in the earlier study) confirms every rewritten value still resolves to a real file via the new resolution path.
RollbackRestore that tenant's pre-migration DB snapshot (already independent per tenant), or run the inverse string-transform if the forward transform is a clean, reversible prefix substitution (recommended: design it to be reversible from the start, so rollback doesn't require a separate backup restore for a same-day catch of a bad row).
DowntimeNone required — because Layer 4's fallback route means old and new values both resolve during the transition window, the UPDATE can run against a live tenant without a maintenance window, as long as it completes inside one DB transaction so no request ever reads a half-updated row.
Data-loss riskThe only real risk is a script bug silently mismatching prefixes on an edge-case value (e.g. an already-absolute-path row from ReceiptVoucher/TaxFiling that doesn't match the expected relative-path pattern). Mitigation: the 2 absolute-path outliers get their own narrower pre-pass (convert absolute→relative first, verified separately) before the common rewrite script runs against all 10 columns uniformly.
Why ReceiptVoucher/TaxFiling need a pre-pass, not the same script

These two store an absolute filesystem path today (Paths.get(uploadDir).toAbsolutePath()), while the other 8 columns store a relative /uploads/... URL. A single rewrite script written for the relative-path pattern would silently no-op (or worse, mismatch) on these two. Treat them as their own migration step, run and validated before the common script, exactly as flagged in the blueprint's Part 03 — this roadmap makes that sequencing explicit rather than implicit.

Part 05

Storage migration strategy — resumable, verifiable, safe

Physical file copy from the flat uploads/{category}/ layout into uploads/{tenant-storage-key}/{category}/, per tenant. Copy-then-verify-then-cutover, exactly as the earlier study specified — this section adds the resume/failure/duplicate handling the task now explicitly asks for.

Process

1. Enumerate. For the tenant being migrated, query all 10 DB columns for every currently-referenced relative path — this is the authoritative "files that matter" list, not a blind directory walk.
↓
2. Copy with a manifest. For each enumerated path, copy (never move) into the new tenant-keyed location, appending one line to a per-tenant migration manifest file (source path, dest path, size, sha256, status: pending/done/failed) as each copy completes.
↓
3. Verify. Compare source and dest checksums for every "done" row. Any mismatch flips that row to failed, logged, not silently retried without record.
↓
4. Resume on re-run. The script is re-entrant: on any interruption (crash, timeout, manual stop), re-running it re-reads the manifest, skips every row already marked done and verified, and only processes pending/failed rows. No file is ever copied twice unnecessarily, and no partial state is lost.
↓
5. Handle missing source files. A DB row referencing a path with no corresponding physical file (possible today — nothing currently guarantees referential integrity between the 10 columns and the actual filesystem) is logged as source-missing in the manifest, not treated as a migration failure — it's a pre-existing data-quality issue this migration surfaces, not one it causes. Recommend a report of these rows delivered to the module owner before cutover, not silently ignored.
↓
6. Handle duplicates. If two DB rows reference the same physical source file (legitimate — e.g. two entities pointing at a shared logo), the copy step is naturally idempotent (copying the same source to the same dest twice is a no-op past the first time) — no special case needed beyond the manifest already deduping by dest path.
↓
7. Cutover gate. Only once every enumerated row is done-and-verified (or explicitly triaged as source-missing and accepted) does that tenant proceed to Migration B (Part 04) and then the grace-window-delayed deletion of old flat-path files.

Failed-migration recovery

Because the manifest is the source of truth for progress (not "did the DB get updated," which only happens after full storage-copy success per the dependency graph), a failed run at any point leaves the tenant's live system completely unaffected — old flat paths are untouched, DB still points at them, nothing has cut over. Recovery is simply: fix the root cause (disk space, permissions, a bad path), re-run the same script, let the manifest's resume logic pick up where it stopped.

Rollback (post-cutover)

Matches the earlier study's Part 13: restore old flat files from the pre-deletion grace-window snapshot, restore the DB rewrite via Part 04's rollback. Because deletion of old files only happens after a grace window (recommend 30 days, matching the soft-delete convention already used elsewhere), a rollback decision doesn't need to be made same-day — there's real time to catch a problem.

Part 06

Security review before implementation — exact enforcement points

The blueprint's Part 08 named the concerns; this roadmap pins each one to the exact class/method where it must be enforced, so it isn't left to individual judgment across 9 call sites.

Single highest-priority security invariant

The tenant segment of every StorageKey must originate only from server-side, per-deployment configuration (Layer 1's tenant.storage-key) — never from a JWT claim, request header, or request parameter that a client could influence. Because each tenant already runs as its own isolated process/DB, this is enforceable simply by reading the config value once at startup and never accepting an override — but it must be reviewed explicitly in code review for every one of the 9 refactored call sites, since a single mistake here (e.g. one developer wiring the tenant segment from a request param "to make testing easier") reintroduces the exact vulnerability class this whole migration is meant to close.

Part 07

Testing strategy

Verified this session: zero existing tests reference any of the 7 current storage classes. This is a green field for test design, not a retrofit — and it means the new abstraction should be built test-first, since there's no legacy test suite to accidentally break.

Backend — unit tests

  • StorageKeyTest — path traversal rejection (`../`, absolute paths, null bytes, encoded traversal sequences), valid-key construction, category/filename edge cases (empty, overlong, non-ASCII).
  • LocalStorageProviderTest — store/load/delete/exists round-trip against a temp directory (JUnit @TempDir); confirms it never writes outside the configured root regardless of what a (theoretically already-validated) StorageKey contains — defense in depth, testing the provider assuming StorageKey's guarantees could somehow be bypassed.
  • TenantFileStorageServiceTest — mocked StorageProvider; confirms tenant segment always comes from injected config, never from a method parameter; confirms MIME/extension/size validation actually rejects bad input before delegating to the provider.
  • Per-call-site tests for all 9 refactored classes (currently zero exist) — at minimum, confirm each still produces a resolvable stored reference after the refactor, using the project's existing Mockito + builder-fixture convention (per CLAUDE.md's documented test style).

Backend — integration tests

  • End-to-end upload→store→retrieve for each of the 9 endpoints (Part 03's table), using @SpringBootTest with a real (test) filesystem root — confirms the full controller→service→provider chain, not just the isolated unit.
  • Cross-tenant isolation test: two simulated tenants (different tenant.storage-key config), confirm tenant A's stored file is never reachable via any manipulation of tenant B's request — this is the single most important new test given Part 06's highest-priority invariant.
  • PosCheckoutControllerTest-style existing-pattern test for the URL-resolution DTO changes (Part 01, Layer 6) — confirm the 4 newly-introduced thin wrappers (CompanyProfile, Employee, SalesOrderAttachment, ReceiptVoucher) serialize correctly and don't leak the old raw-entity shape to any existing frontend consumer.

Backend — migration tests

  • DB rewrite script (Part 04) tested against a seeded copy of a representative tenant DB — assert row counts unchanged, assert idempotency by running the script twice and confirming the second run is a no-op.
  • Storage migration script (Part 05) tested against a seeded temp directory with deliberately-injected edge cases: a missing source file, a duplicate reference, a mid-run interruption (kill and resume) — confirms the manifest-driven resume logic actually resumes correctly rather than re-copying or skipping incorrectly.

Frontend

  • Image loading: manual + automated smoke check that getImageUrl() output still renders for all ~35 consuming pages after backend URL-resolution changes (Layer 6) — no frontend code change expected, so this is a regression check, not new test development.
  • Upload flow: exercise each of the 9 upload UI flows (product image, brand logo, company logo/stamp, branch logo/stamp, employee avatar, own-profile avatar, sales order attachment, receipt voucher attachment, tax filing attachment) end-to-end in a running dev instance, confirming success and error states both render correctly against the refactored backend.
  • Existing UI compatibility: because Product's image handling was found to be different from what a naive read of the entity suggests (separate ProductMedia table, dead ProductResponse.image field), specifically verify the Products page's image display end-to-end before assuming any change there is safe.

Infrastructure

  • Kubernetes: deploy the refactored backend into the existing k8s/max pilot namespace; confirm the tenant-keyed subfolder appears correctly under the existing PVC mount (kubectl exec ... -- ls /app/uploads); confirm the Recreate deployment strategy still behaves correctly (no split-brain writes during a rollout).
  • Docker: confirm the new explicit VOLUME /app/uploads declaration doesn't change existing local-dev behavior for anyone running the backend via plain mvn spring-boot:run (no Docker) — this is additive Dockerfile metadata, should be a no-op outside a container.
  • Volume mounts: confirm a pod restart correctly re-mounts the same PVC and the tenant-keyed subfolder survives — direct extension of the K8s pilot's already-proven PVC durability, just re-verified with the new subpath in place.
  • Backup/restore: dry-run a full backup-then-restore of one tenant's uploads PVC subfolder into a scratch environment, confirming file integrity (checksum match) end to end — this exercises Part 05's design of Blueprint's Part 05 (Backup & DR) for the first time against real infrastructure, not just in the abstract.

Migration tests (full rollout rehearsal)

  • Existing tenant: full dry run against a cloned copy of one live tenant's DB + uploads (recommend demo — lowest risk, matches the original report's Phase 4 choice) — the complete Part 04 + Part 05 process end to end, followed by the validation checklist.
  • New tenant: provision a brand-new tenant from scratch using the post-refactor code path (no legacy flat-path files to migrate at all) — confirms the new-tenant onboarding path is clean and doesn't accidentally depend on migration-specific code.
  • Rollback: deliberately trigger a rollback at each of the three stages defined in the earlier study's Part 13 (after copy, after DB rewrite, after old-file deletion) against the scratch tenant, confirming each rollback path actually restores a working system.
  • Export: exercise the blueprint's Part 04 export flow against the migrated scratch tenant, confirm the manifest + checksums are correct.
  • Import: import that export into a second scratch environment under a different tenant-storage-key (proving the clone/rename capability Part 04 of the blueprint specifically called out as "free" once Layer 1/2 are correct) and confirm collision detection correctly rejects an import attempt that reuses an existing key.

Part 09

Final implementation phases

Nine phases, each independently deployable where the dependency graph (Part 02) allows it. Phases 1-4 build the foundation with zero effect on production data; Phase 5 is the one true data-migration phase, run per-tenant; Phases 6-9 are the blueprint's previously-deferred capabilities, cleanly separable.

Phase 1 — Tenant Storage Configuration

Objective: establish a stable, immutable identifier per tenant, independent of Spring profile naming.
Files to modify: all 14 application-{tenant}.properties files (add tenant.storage-key=...); remove dead upload.path from base + 4 tenant files.
Expected output: every tenant profile carries a unique, permanent storage key.
Validation checklist: manual cross-check that all 14 keys are distinct (no automated uniqueness check possible across independent processes); confirm no code currently reads the removed upload.path (already verified — zero references).
Rollback: revert the properties files; no code or data affected.
Completion criteria: 14/14 tenants have a reviewed, unique storage key committed.

Phase 2 — Storage Abstraction Core

Objective: ship StorageProvider, StorageKey, LocalStorageProvider, TenantFileStorageService — fully tested, zero existing call sites touched yet.
Files to modify: new files only (Part 01's "new components" table).
Expected output: a working, independently testable storage layer with no production behavior change (nothing calls it yet).
Validation checklist: full unit test suite (Part 07) green; manual traversal-attack test against StorageKey.
Rollback: trivial — delete the new files; nothing else in the system references them yet.
Completion criteria: code review approved with explicit sign-off on the Part 06 security checklist.

Phase 3 — Metadata Table + Backward-Compat Route

Objective: add stored_file (Flyway V43) and the StaticResourceConfig fallback handler — both additive, both independently deployable, can ship together or separately.
Files to modify: db/migration/V43__stored_file.sql, common/storage/StoredFile.java, StoredFileRepository.java, config/StaticResourceConfig.java.
Expected output: new table exists in all 14 tenant DBs (via normal Flyway boot); old flat-path URLs still resolve (unchanged) alongside a new (currently unused) fallback route ready for Phase 5.
Validation checklist: Flyway migration runs clean on every tenant DB on next boot; confirm old /uploads/** URLs still resolve exactly as before (regression check, not a new behavior).
Rollback: Flyway migration is additive-only — safe to leave in place even if the rest of the effort pauses; fallback route removal is a one-line revert if needed.
Completion criteria: table present and empty (no writer yet) in all tenant DBs; static resource regression test green.

Phase 4 — Call-Site Refactor + DTO Mapping (Layers 5-6)

Objective: migrate all 9 upload call sites onto TenantFileStorageService; introduce URL resolution at the 9 DTO/mapping points (4 requiring new wrapper classes).
Files to modify: the full "existing components that change" + "DTO/response mapping" tables from Part 01 — 9 storage call sites, 5 existing mappers edited, 4 new wrapper classes.
Expected output: new uploads for all 9 flows are written under uploads/{tenant-storage-key}/{category}/ and stored in the DB as logical keys; existing (pre-Phase-4) uploads are unaffected until Phase 5 migrates them.
Validation checklist: full integration test suite (Part 07) green, including the cross-tenant isolation test; manual smoke test of all 9 upload UI flows; confirm existing (unmigrated) files still render via the Phase-3 fallback route.
Rollback: per-call-site revert is possible since each of the 9 refactors is an independent commit — a regression in one doesn't require rolling back all 9.
Completion criteria: all 9 call sites merged, tested, and deployed; new uploads verified end-to-end in a live (non-production) environment.

Phase 5 — Per-Tenant Data Migration (Layers 7-8-9, repeated per tenant)

Objective: migrate one tenant's existing files + DB rows to the new scheme, validate, cut over.
Files to modify: none (this phase runs migration scripts and SQL, not application code changes) — recommend demo first, then the remaining 13 tenants one at a time.
Expected output: that tenant's uploads/{tenant-storage-key}/ populated and verified; that tenant's 10 DB columns rewritten to logical keys; old flat-path files retained through the grace window, then deleted.
Validation checklist: full Part 04 + Part 05 validation (row counts, checksums, HTTP-fetch sample, manual UI spot-check across all 9 upload types) for that specific tenant.
Rollback: per Part 04/05's three-stage rollback design, available up until the grace window closes for that tenant.
Completion criteria: that tenant fully validated and cut over; repeat for the next tenant. Independently deployable per tenant — a problem discovered on tenant 3 doesn't block or need to unwind tenants 1-2.

Phase 6 — Storage Maintenance

Objective: orphan detection, cleanup sweep, duplicate report, usage statistics, folder health check (blueprint Part 07).
Files to modify: new scheduled-job classes following the confirmed PosTerminalAutoArchiveJob idiom.
Expected output: scheduled jobs running in report-only mode initially (no auto-delete until confidence is established).
Validation checklist: first several runs manually reviewed before enabling any auto-delete behavior.
Rollback: disable the @Scheduled job (config toggle); no destructive action has occurred by default.
Completion criteria: at least one full report-only cycle reviewed and judged accurate before any auto-cleanup is enabled.

Phase 7 — Export / Import Tooling

Objective: blueprint Part 04's export/import capability, built entirely on the now-stable Phase 1-2 abstractions.
Expected output: a working export-one-tenant / import-into-new-environment tool, tested against the scratch tenant from Part 07's migration-test rehearsal.
Validation checklist: full round-trip (export → import into a differently-keyed environment → validate) exercised at least once against a non-production tenant.
Completion criteria: documented, tested export/import runbook exists.

Phase 8 — Backup / DR Automation

Objective: blueprint Part 05's independent per-tenant, per-asset-class backup schedule and restore-drill automation.
Expected output: scheduled backups running; at least one automated restore drill completed successfully.
Completion criteria: a documented, tested DR runbook, with at least one full restore-drill success on record.

Phase 9 — (Deferred) Object Storage Swap-In

Objective: not scheduled as part of this rollout — tracked as a future initiative, triggered only if/when multi-replica backend scaling becomes an actual operational requirement, per the blueprint's explicit deferral (Part 09).

Part 10

Architecture validation — critical re-review of the blueprint itself

Before converting the blueprint into these phases, it was re-examined for weaknesses that only surface once you try to sequence real implementation work against it. Most of it holds up. Three points are worth surfacing explicitly, none of which are severe enough to block starting Phase 1.

Hidden assumption found: not every entity has a mapper

The blueprint's Part 03 assumed a uniform "insert one line in the DTO mapper" cost for the URL-resolution step. Verified this session: 4 of 9 entities (CompanyProfile, Employee, SalesOrderAttachment, ReceiptVoucher) return raw JPA entities today with no mapper at all. This isn't a flaw in the blueprint's direction — Option C is still correct — but Phase 4's effort estimate needs to account for 4 new wrapper classes, not just line edits. Reflected in Part 01/09 above; no architecture change needed, just a corrected estimate.

Hidden assumption found: Product's image handling doesn't match a naive read of the entity list

Both the original audit and the blueprint listed "ProductMedia.imageUrl" as if it were a straightforward peer of Brand's logoPath or Branch's logoUrl. In practice, product images live in a genuinely separate entity/repository (ProductMedia), and a differently-named, unrelated class (ProductResponse.image) is dead code that could mislead a developer skimming for "the" product image field. Recommend Phase 4 include a short, explicit design note confirming how ProductMedia rows map into the response before writing that one call site's DTO resolution — a small scoping task, not an architecture concern.

Scalability concern: none new, previously-identified constraint stands

The ReadWriteOnce PVC / single-replica-per-tenant backend constraint (blueprint Part 07) is unchanged by this roadmap and remains correctly deferred to a future object-storage phase (Phase 9, not scheduled). No new scalability concern surfaced in this review — the phased plan above doesn't make this constraint worse or better; it simply doesn't touch it, as intended.

Maintainability: verified as an improvement, not a new concern

Collapsing 9 independent hardcoded implementations into one abstraction, backed by a previously-nonexistent test suite (all-new per Part 07), is a maintainability improvement over the current state by every measure re-checked this session — no maintainability regression identified.

Kubernetes concern: none new

Re-confirmed: no structural K8s change required: existing PVC, mount point, Recreate strategy, and ingress rules all remain valid under the new subpath convention. This was true in the original blueprint and remains true after this review.

Multi-client concern: the one genuinely load-bearing risk, already flagged, worth restating

The entire architecture's tenant-isolation guarantee rests on Phase 1 being done correctly and reviewed carefully (Part 08's highest-risk item). This isn't a weakness in the blueprint's design — it's an inherent property of any tenant-keyed system — but it deserves to be named plainly here: the security of this whole redesign is only as strong as the discipline applied to provisioning and reviewing tenant storage keys. No architecture change mitigates this further than what's already specified (Part 06); it is a process discipline, and the roadmap accounts for that by making it its own reviewed phase rather than a config afterthought.

Future cloud migration concern: none new

Re-confirmed against the layered StorageProvider design: every forward path named in the blueprint's Part 09 (CDN, object storage, multi-region, horizontal scaling, ReadWriteMany, hybrid) remains reachable through config/new-provider-implementation alone, with no business-logic change, exactly as designed.

Verdict

The blueprint is implementation-ready. Nothing found in this review requires a change to the architecture itself — only two effort-estimate corrections (DTO wrapper count, Product's image-handling scoping) and one restated (not new) emphasis on Phase 1's process discipline. Proceed to Phase 1 as specified.

Part 11

Final Go / No-Go decision

Decision

GO. Implementation should begin immediately, starting with Phase 1. No further architecture changes are required before starting. This roadmap's Part 10 review confirms the blueprint is sound; the module inventory (Part 03) confirms scope is fully bounded (9 call sites, not an open-ended search); the dependency graph (Part 02) confirms a safe, non-circular build order; and the risk matrix (Part 08) confirms every identified risk has a concrete mitigation and recovery path, with the single highest-risk item (Phase 1 key provisioning) being a process control that can be enforced starting on day one.
