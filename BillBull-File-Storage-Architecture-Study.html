File Storage & Multi-Tenant Upload Architecture

A ground-truth audit of every place BillBull writes an uploaded file, why the current flat uploads/ layout blocks independent client migration, and a production-grade redesign that makes one tenant's files exportable, importable, and disposable without touching anyone else's.

Branch feature/billpos Scope backend, frontend, schema, K8s manifests — 1 research pass Status no code changed, no data moved Prepared 2026-07-23

Part 01

Current architecture, documented ground-truth

Every uploaded file — product image, logo, stamp, avatar, attachment — lands under one relative directory, ./uploads/, resolved against the JVM's working directory. There is no tenant identifier anywhere in the write path. Isolation between clients today exists only because each tenant happens to run as its own OS process (or, on the new k3s pilot, its own pod with its own PVC) — never because the application code segments paths by client.

The one static resource mapping

config/StaticResourceConfig.java registers exactly one handler for the whole backend:

registry.addResourceHandler("/uploads/**")
        .addResourceLocations("file:./uploads/");

Every uploaded asset, regardless of module, is served back to the frontend under this single /uploads/** URL prefix, resolved to ./uploads/ relative to wherever the JVM was launched from.

upload.path — configured, but dead

Finding

The base application.properties defines upload.path=uploads/ (line 85), and four of the fourteen tenant profile files (demo, leroyalflowers, leroyalgifts, royaltools) override it with what looks like a tenant-specific subfolder, e.g. uploads/leroyalflowers/. A repo-wide search of src/main/java found zero classes that read or inject this property. It is orphaned configuration — even those four tenants get no actual namespacing effect from it. Whoever wrote those four overrides was reaching for exactly the isolation this report recommends, half a step ahead of the code that would have honored it.

A second, unrelated property key, file.upload-dir, appears as a constructor default in two services (TaxService.java:46, ReceiptVoucherService.java:63) but is never set in any properties file either, so both always fall through to their hardcoded defaults (uploads/tax_filings, uploads/receipts) — identical across every tenant.

There is no shared storage abstraction. Each module either hardcodes its own directory constant or writes inline in the controller. All eight patterns coexist side by side:

No tenant identity exists in the request pipeline

Tenancy in BillBull today is "one Spring Boot process, one Postgres database, selected by spring.profiles.active" — full stop. There is no TenantContext, no CompanyContext, no client code anywhere in the JWT claims or security context that upload code could consult. The only per-request context holder that exists is security/BranchContextHolder, and that's intra-tenant branch scoping (multiple outlets inside one client's single DB) — a different axis entirely, and one worth not confusing with the client/tenant axis this report is about.

This matters for the redesign: there is currently nothing to key a path on except the Spring profile name itself, which is exactly what four tenant properties files already guessed at (and what the recommendation in Part 9 formalizes properly).

Part 03

Root cause analysis

The flat layout isn't a bug in any one class — it's the absence of a decision. Four compounding root causes:

  1. No storage abstraction ever existed. Each module that needed to save a file wrote its own three-line "resolve dir, create if missing, write" logic, independently, over the life of the codebase. Nine call sites, nine copies, zero shared interface — the same failure pattern already documented for reservation logic in the companion stock study (duplication instead of generalization).
  2. Tenancy is external to the app, not modeled inside it. Because "which tenant is this?" is answered entirely by which properties file activated the process — never by a value available inside a request — no upload code had a tenant key to segment by even if someone had wanted to.
  3. The one config knob that could have fixed it was never wired up. upload.path exists, four tenants even set it distinctively — but nothing reads it. This is the closest the codebase ever came to solving this problem, and it silently didn't work.
  4. Multi-tenant deployment concerns arrived after the storage code was written. The single-tenant-per-process model made a flat uploads/ harmless for years — each tenant's process had its own working directory, so files never actually mixed until multiple tenants started sharing one host/filesystem (confirmed by the k3s RUNBOOK.md: four legacy tenants previously shared one systemd WorkingDirectory, meaning their uploads/ trees were commingled on the same disk before any K8s migration began). The problem was latent, then became active the moment infrastructure consolidated.
Corroborating evidence from the K8s pilot

The existing k8s/max/RUNBOOK.md (client2's completed 2026-07-16 migration) explicitly notes that tenant "max" had zero real uploads at cutover, so the uploads-migration step was skipped — and separately flags that the next tenants queued for migration "likely do have real data" and will need a real upload-directory migration. This report exists to answer exactly that flagged need, for every remaining tenant, once and for good rather than tenant-by-tenant.

Part 04

Impact analysis — every affected surface

Anything that reads Part 02's table is affected. Grouped by blast radius:

Backend storage classes (7)

FileUploadUtil, EmployeeServiceImpl, ProductImageStorageService, BrandLogoStorageService, CompanyProfileService, BranchService, ReceiptVoucherService, TaxService, plus the inline SalesOrderController writer — all need their directory-resolution logic replaced by one shared, tenant-aware component.

StaticResourceConfig

The single /uploads/** → ./uploads/ mapping must be revisited so old URLs keep resolving during transition (see Part 10 backward-compat routing).

10 DB path/URL columns

Every stored path (Part 02 table) either needs a migration to a new value or a URL-rewrite layer that keeps old values valid (Part 12).

Frontend: single choke point

getImageUrl() in urlUtils.js is the only place that builds a browser-facing URL from a stored path — ~35 consuming pages need zero changes if new paths still resolve under /uploads/**.

K8s manifests

k8s/*/pvc.yaml, deployment.yaml volume mounts — no structural change needed (see Part 07), but backup/DR tooling built against "the PVC = the tenant" assumption should be checked against the new subpath convention.

Backend Dockerfile

No VOLUME /app/uploads declared today; worth adding explicitly once the path convention is finalized, for local/dev parity with the K8s mount.

Two stub/edge paths

Quotation attachments (DB BLOB, no disk path — unaffected either way) and Salary Advance attachments (currently a no-op stub) should be decided on: fold into the new scheme now, or explicitly deferred and tracked.

Every module named in the original ask — product images, company/branch logos and stamps, customer/supplier/employee images (customer and supplier image upload were not found as implemented endpoints in this audit — see caveat below), user avatars, attachments, POS assets, receipt logos — maps onto the table in Part 02 with one exception worth flagging explicitly.

Scope caveat

This audit found no dedicated upload endpoint for customer images or supplier/vendor images in the current backend — only Excel import endpoints exist for those two entities, and those parse in-memory without persisting a file. If customer/supplier photo upload is planned or exists in a part of the codebase this pass didn't surface, it should be added to Part 02's table before implementation begins; the storage abstraction in Part 09 is designed to make adding a new upload type a one-line registration, not a new copy of the pattern.

Part 06

Storage strategy comparison

Four options, scored against the criteria the task specifies. All four assume the same in-app fix regardless of backend choice: a shared storage abstraction that takes a tenant key, a category, and a file — the physical backend (local disk vs. object store) becomes an implementation detail behind one interface.

Option A — uploads/{client}/ flat per-client

Every tenant gets one top-level folder; all its files (of every type) sit inside it undifferentiated.

CriterionAssessment
Client migrationSolved — one folder = one client, trivially tarball-able
K8s compatibilityFine — still local disk, still one PVC per tenant pod
MaintainabilityWeak — no per-category structure inside the client folder, same nine hardcoded subfolder names collide again one level down
Future cloud migrationRequires a second migration later to add category structure

Option B — uploads/{tenant-code}/{category}/ (recommended local layout)

Client-level isolation and category structure in one scheme — the natural evolution of what four tenant properties files already half-attempted with upload.path.

CriterionAssessment
Client migrationSolved — copy uploads/{tenant-code}/, done
K8s compatibilityExcellent — no infra change; still one PVC per tenant, mounted at the same path
Backup / DRPer-category selective restore possible (e.g. restore only products/ without touching logos/)
PerformanceIdentical to today — local disk I/O
Security / isolationFilesystem-level only; no cryptographic tenant boundary, but matches the trust model already in place (one process per tenant DB)
Future cloud migrationThe tenant-code/category prefix maps directly onto an S3 key prefix later — zero rework of the path convention, only the storage backend swaps out
Ease of deploymentHighest of all four — no new infrastructure, no new dependency, ships as an app-only change

Option C — Object storage (MinIO/S3-compatible) with tenant-prefixed keys

CriterionAssessment
Client migrationSolved, and more elegantly — prefix-based export/import via bucket replication or mc mirror, no filesystem access needed at all
K8s compatibilityBest — removes the ReadWriteOnce PVC constraint entirely, so the backend can finally run RollingUpdate with >1 replica instead of the current Recreate/single-replica limitation the K8s handbook documents
Scalability / HAMinIO itself can be deployed as a multi-node erasure-coded cluster; trivially horizontally scalable
PerformanceAdds one network hop per file read/write versus local disk — usually sub-10ms in-cluster, negligible for this workload (receipts, product images, logos), but non-zero
SecurityPer-bucket or per-prefix IAM policy gives a real access boundary, stronger than filesystem permissions
Ease of deploymentNew moving part — MinIO deployment, credentials, bucket lifecycle policy, backend SDK dependency, code changes to every one of the 9 write sites
Future cloud migrationZero rework — MinIO's S3 API is what AWS S3 speaks; swapping the endpoint is a config change, not a code change

Option D — Hybrid: local disk (Option B layout) now, object storage later

Adopt the Option B path convention immediately (cheap, ships fast, solves the migration problem today) while designing the storage abstraction's interface so that swapping the backend to MinIO/S3 later touches only one class, not nine call sites.

CriterionAssessment
Time to valueFastest — Option B alone is a contained, low-risk change deployable in the current architecture
Future-proofingThe tenant-prefixed key convention designed for Option B is identical to the S3 key convention Option C would use — no wasted work, no second migration of the path scheme, only of the storage backend if/when K8s scale-out demands it
RiskLowest — no new infrastructure dependency introduced in the phase that actually solves the stated problem (independent client migration)
Recommendation preview

Option D. Solve the actual stated problem — a client's files trapped inside a shared flat directory — with Option B's tenant/category path convention now, behind a storage-abstraction interface. Defer Option C (object storage) as a backend swap-in for whenever multi-replica horizontal scaling of the backend actually becomes a requirement, since today's single-tenant-per-pod K8s model doesn't yet need it. Full reasoning in Part 09.

Part 07

Kubernetes readiness

The existing k8s/max/ pilot already answers most of this — worth restating precisely what it already solved versus what remains open, since the two are easy to conflate.

Already solved by the existing K8s pilot

  • Persistent Volumes. k8s/max/pvc.yaml provisions a 10Gi ReadWriteOnce PVC (local-path storage class) per tenant, mounted at /app/uploads in that tenant's Deployment — durable across pod restarts, unlike the container's ephemeral filesystem.
  • Per-tenant isolation at the volume level. Each tenant Deployment gets its own dedicated PVC — "max" (client2) has max-uploads-pvc; a new tenant gets its own named PVC. Tenant A's pod cannot mount Tenant B's volume.
  • Pod replacement / self-healing. Standard Deployment/ReplicaSet reconciliation already covers this — a crashed pod is replaced and re-mounts the same PVC.

The real constraint: ReadWriteOnce forces single-replica, Recreate-only

Because the uploads PVC is ReadWriteOnce, only one pod can hold it at a time. This is why the backend Deployment is pinned to replicas: 1 and uses the Recreate strategy (kill old pod, then start new) instead of RollingUpdate (start new pod before killing old — briefly two pods, both wanting the volume). The handbook is explicit and correct about this trade-off: it accepts brief downtime on every deploy in exchange for never risking split-brain writes to the shared disk.

Not solved by this report's recommendation, by design

Adopting Option B's tenant/category path convention (Part 06) does not remove the RWO constraint — it's still one PVC, one writer. If horizontal scaling of a single tenant's backend (multiple replicas serving one client) becomes a real requirement, that specifically requires Option C (object storage) or a ReadWriteMany-capable storage class (e.g. NFS-backed, EFS on AWS) — a separate decision from the one this report is making. Flagging this now so it isn't rediscovered as a surprise later.

What does need to change for K8s

  • Nothing structural. Option B's uploads/{tenant-code}/{category}/ layout lives entirely inside the existing mount point (/app/uploads) — the PVC, the volume mount, the Recreate strategy, the ingress rule routing /uploads to the backend Service all stay exactly as they are.
  • Backend Dockerfile should gain an explicit VOLUME /app/uploads declaration for local/dev-Docker parity with the K8s mount — currently absent, harmless in K8s (the PVC mount overrides it anyway) but worth adding for consistency.
  • Backup tooling (whatever job snapshots the PVC — not found as existing automation in this repo, likely a manual/host-level `kubectl cp` or volume-snapshot process today) should be aware of the new per-tenant-code subfolder so a "restore just this tenant's uploads" operation can target the correct in-volume path, not just the whole PVC.
  • High availability for the backend tier specifically remains bounded by the RWO PVC regardless of this report's recommendation — see the callout above. HA for the frontend tier is already unaffected (stateless, replicas: 2 today).

Part 08

Client migration strategy

The goal: move Tenant A to a different database/server/cluster touching only Tenant A's data — database and uploads both — with zero risk to any other tenant's files.

1. Export Tenant A's database (existing per-tenant Postgres dump — already independent today, no change needed)
↓
2. Export Tenant A's uploads: tar -czf {tenant-code}.tar.gz uploads/{tenant-code}/ — a single directory tree, self-contained, no cross-tenant references possible because DB path columns store /uploads/{tenant-code}/...-relative values that never point outside that tree
↓
3. Transfer both artifacts to the destination environment (new server, new K8s namespace, new PVC)
↓
4. Restore DB dump into the new Postgres instance; extract the uploads tarball into the destination's /app/uploads/{tenant-code}/ (same relative subpath, so every stored path column resolves without a single UPDATE)
↓
5. Point the new Spring profile / ConfigMap at the restored DB; boot; verify a sample of images/logos/attachments render

Step 2 and step 4's subpath match is what makes this zero-touch on the database — because the tenant-code segment is already the first path component under the shared root, moving the whole subtree preserves every relative URL stored in every DB column untouched. This is the direct payoff of choosing tenant-code as the top-level segment (Option B) rather than nesting it deeper.

Why this beats today's alternative

Today, the equivalent operation requires copying the entire shared uploads/ directory (every tenant's files) and then manually identifying and stripping out everything that doesn't belong to Tenant A — file-by-file, with no directory boundary to lean on, and no way to be fully sure the DB's absolute-path columns (ReceiptVoucher, TaxFiling) will even still resolve on the new host. The new layout makes this a single tar command over one subtree, matching the DB export's own independence.

Part 09

Recommended final architecture

Local disk, Option B layout (uploads/{tenant-code}/{category}/), behind one shared storage-abstraction service, with the tenant code resolved from the active Spring profile — no new infrastructure, no new dependency, ships as an application-only change. Object storage (Option C) is documented as the deliberate next step, not adopted now, because nothing in the current single-replica-per-tenant K8s model needs it yet, and introducing MinIO today would add an operational dependency to solve a problem (horizontal scale-out) that doesn't exist yet.

Why this beats the alternatives, specifically

  • vs. Option A (client-only, no category folders): B avoids re-colliding all nine hardcoded subfolder names one level down inside each client folder; category structure makes per-type backup/retention policy possible later (e.g. purge old receipt attachments on a schedule without touching product images).
  • vs. Option C (object storage now): B ships without a new service to deploy, secure, and operate; it fully solves the stated problem (independent client migration) with dramatically less risk and time; and because the path convention is chosen to be S3-key-compatible, adopting C later is a backend swap, not a redesign.
  • vs. doing nothing: the RUNBOOK.md already flags that upcoming tenant migrations "likely do have real data" — this is not a hypothetical future problem, it's the literal next migration in the queue.

Tenant code source

Use the active Spring profile name itself (client1, demo, geebu, etc. — the same string already in SPRING_PROFILES_ACTIVE) as the tenant-code path segment. No new identity concept is introduced; it's the one tenant key that already exists and is already unique per deployed process. A single new @Value("${spring.profiles.active}")-backed bean (or reuse of Environment.getActiveProfiles()[0]) is the only "tenant context" this recommendation requires — deliberately not building a general-purpose multi-tenant-in-one-process framework, since that isn't how BillBull is deployed and isn't in scope.

The storage abstraction

One new component, e.g. common.storage.TenantFileStorageService, with a signature shaped like:

String store(String category, MultipartFile file)   // returns "/uploads/{tenant}/{category}/{uuid}.{ext}"
Resource load(String relativeUrl)
void delete(String relativeUrl)

All nine existing call sites (Part 02) are refactored to call this instead of their own hardcoded directory logic — collapsing seven duplicated implementations into one. This is the same "generalize the duplicated concept instead of leaving parallel copies" fix already recommended for PosStockReservation vs. BatchAllocation in the companion reservation study; the same anti-pattern shows up here in a different subsystem.

Frontend impact

Part 11

Ten columns across nine entities (Part 02 table) hold a stored path. All but two store a value that's already relative to the uploads/ root and prefixed with /uploads/ — those need only a string segment inserted after /uploads/ and before the existing category. Two are the outliers.

Part 12

Zero-data-loss migration plan

Copy-verify-cutover, never move-then-hope. The existing flat directory is never deleted until the new layout is confirmed serving traffic correctly for that tenant.

Step 1 — Snapshot. Back up the tenant's current uploads/ directory and take a DB backup, before touching anything.
↓
Step 2 — Copy, don't move. For each existing category folder, copy (not move) its contents into uploads/{tenant-code}/{category}/. The old flat paths remain live and fully functional throughout.
↓
Step 3 — Rewrite DB path columns. Run the guarded UPDATE script from Part 11 against a copy of the DB first (or inside a transaction with a pre-commit row-count sanity check), inserting the tenant-code segment into every stored path.
↓
Step 4 — Backward-compatible fallback route (transition safety net). Temporarily extend StaticResourceConfig with a second handler serving the old flat paths (/uploads/{category}/**, distinguishable from the new /uploads/{tenant-code}/{category}/** because tenant codes are a known, enumerable set) so any URL cached client-side, in an email, or in a previously-generated PDF still resolves during the transition window.
↓
Step 5 — Validate. Automated script fetches every rewritten path via HTTP and confirms 200 + non-zero content-length for a sample (or all) rows per affected table; manual spot-check of product images, a company logo, and one receipt attachment through the actual UI.
↓
Step 6 — Cutover. Only after Step 5 passes for a tenant: remove that tenant's old flat-path files and retire the Step 4 fallback handler for that tenant (or leave the fallback in place permanently if the cost of doing so is negligible — it's a static resource handler, not a runtime branch).

Batch migration script shape

One script, parameterized by tenant code, run once per tenant (the whole client fleet is 14 tenants — this is a bounded, enumerable, one-time job, not an ongoing process):

for each tenant in {client1, client2, ..., royaltools}:
    rsync -a  uploads/{category}/  →  uploads/{tenant-code}/{category}/   (per category, filtered by that tenant's known file list from DB)
    run  V{next}__namespace_upload_paths.sql   against that tenant's DB only
    run  validate_uploads.sh {tenant-code}      (HTTP HEAD every path in the 10 columns, report any 404)
    on all-green: mark tenant migrated; old flat files retained N days before deletion (grace window, not immediate delete)

Because each tenant already lives in its own database and (per the K8s pilot) increasingly its own filesystem/PVC, tenants can be migrated one at a time, independently, with no coordination required between them — matching exactly the independence this whole report is trying to establish for the running system, not just the tooling that migrates it.

Backward compatibility during transition

Step 4's fallback handler is the key device: nothing breaks between "DB rewritten" and "old files deleted," because both the old and new physical files, and both the old and new resolvable URLs, coexist during that window. This is the same category of caution the project already uses for schema changes (Flyway scripts guarded with to_regclass checks) applied to a filesystem/URL migration instead of a schema migration.

Part 13

Rollback strategy and validation

StageRollback action
After Step 2 (files copied, DB untouched)Delete the new {tenant-code}/ tree; nothing else changed. Zero risk — trivial rollback.
After Step 3 (DB rewritten)Re-run the inverse REPLACE (strip the tenant-code segment back out) from the pre-migration DB backup, or restore that tenant's DB backup directly since each tenant's DB is already fully independent of every other tenant's.
After Step 6 (old files deleted)Restore old flat files from the Step 1 snapshot; re-run the Step 3 rollback. This is why Step 6 enforces a grace window before deletion rather than deleting immediately on validation pass.

Validation checklist per tenant

  • Row counts in each of the 10 path columns match before/after the UPDATE (no rows silently skipped or double-rewritten).
  • Automated HTTP fetch of every rewritten path returns 200 with expected content-type/length.
  • Manual UI smoke test: view a product with an image, view Company Settings logo/stamp, view a Branch logo, open one Receipt Voucher attachment, open one Tax Filing attachment, view an employee avatar, view own profile avatar.
  • Confirm the K8s PVC mount for that tenant shows the new {tenant-code}/ subfolder present and correctly populated (kubectl exec ... -- ls /app/uploads).
