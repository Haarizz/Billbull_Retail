Stock Reservation Architecture in BillBull ERP

A ground-truth trace of every place the system holds, computes, or forgets to hold stock — from POS scan to warehouse ledger — prepared to inform a redesign decision.

Branch feature/billpos Scope backend, frontend, schema, 3 research passes Status no code changed Prepared 2026-07-22

Part 1

Functional reservation flows

Seven document types were checked against the codebase, not assumed. Three (POS Layaway, Sales Order, Sales Invoice) write hard reservation rows for batch-controlled lines; two (Quotation, Proforma) compute a live "soft" reservation from document status at query time; two (Stock Transfer, GRN/LPO) never reserve at all.

POS Layaway / Hold

The only flow with a genuinely dual-mechanism reservation: batch-controlled lines go through BatchSelectionService (pin a scanned unit, or FEFO-auto-reserve the line quantity); non-batch lines go through the new PosStockReservationService, a warehouse-level soft hold with no batch/bin identity. Both are invoked from the same loop in PosLayawayService.create(), and both are released together on every exit path — cancel, void, expire, manual release, convert. A "Hold" is the identical code path with depositRequired=false; there is no separate hold entity.

Verified detail

A daily job, PosLayawayExpiryReleaseJob (cron 0 30 2 * * *), sweeps layaways past their due date and calls the same releaseExpired() path. The layaway's displayed status flips to EXPIRED at read time via a computed getter; the job's only job is to stop the stock hold from silently understating availability forever. It is idempotent by construction.

Sales Order

Confirming an order with batch-controlled lines requires an exact BatchAllocation match — either FEFO auto-pick or a manual pick via the same modal used by Delivery Note and Sales Invoice. Non-batch lines are only availability-checked, never held; their contribution to "reserved" everywhere in the app is a live JPQL SUM over open, non-invoiced sales-order lines, computed fresh on every read.

Sales Invoice — a third hard-reservation path, easy to miss

Sales Invoice reserves stock through the same BatchSelectionService engine as Sales Order and Delivery Note, tagged sourceDocumentType=SALES_INVOICE — it is not a pass-through of the Sales Order's reservation, it creates its own. Three distinct entry points feed it, matching the three ways an invoice line can be filled: manual pick (saveSalesInvoiceLineSelection, only on Direct-Sale invoices — Workflow-Driven invoices instead inherit batches from their Delivery Note and never call this), auto-FEFO at post time (autoReserveFefoForSalesInvoiceLine, the Fast-Sale path, since Fast Sale has no manual-picking step at all), and scanned-unit pin (reserveScannedBatchForSalesInvoiceLine, POS unified-scan — the physical scan is treated as the authorization, bypassing the manual-select permission check). Non-batch lines on an invoice are not held by this mechanism; they fall back to the same live-query "soft" contribution described for Sales Order below.

Release happens on invoice delete, on invoice cancel, and — notably — when a linked Sales Order transitions to INVOICED: at that point the SO-level reservation is explicitly released (releaseSalesOrder) because the invoice's own reservation, and eventually its auto-generated Delivery Note's consumption, now owns the stock. This is the one clean, explicit hand-off in the whole system — everywhere else (Proforma→DN, SO→DN when linked directly) the "hand-off" is implicit, done by a query excluding the older document rather than an explicit release call.

Quotation & Proforma — soft only

Neither document writes a reservation row anywhere. QuotationItem.java carries an explicit developer comment stating stock flags are resolved at read time precisely because quotations don't reserve. Their contribution to system-wide "available" is a live sum over APPROVED/ISSUED documents; insufficient stock at approval only logs a warning — the code literally names this the "Soft Reservation Rule." Proforma's query explicitly excludes any proforma with an active Delivery Note against it, which the source comment describes as the DN "taking over" the reservation — a hand-off in effect, not in data.

Delivery Note — the actual pivot

This is where reservation becomes physical. For SO-sourced batch lines, DN does not allocate fresh — it looks up the SO's existing RESERVED rows and flips them to CONSUMED at markDelivered(), posting one outbound StockMovement per allocation. For non-batch lines (and for batch lines the DN itself originates), it performs a genuine FEFO walk over bin stock identities at delivery time and deducts directly. Cancelling a delivered DN reverses both: StockMovement is reversed and consumed allocations return to RESERVED.

Stock Transfer, GRN/LPO — no reservation concept

Transfer posts an immediate negative StockMovement at source on Send and an immediate positive one at destination on Receive — a two-leg debit/credit, confirmed by the complete absence of any BatchAllocation/PosStockReservation import in the 647-line service. In-transit stock is simply absent from both ledgers; there is no in-transit hold bucket. GRN/LPO never reserve expected incoming stock against a downstream sales order — grepped clean.

Part 2

Backend components

Everything that creates, releases, or consumes a reservation funnels through exactly two services. Nothing else constructs a BatchAllocation or PosStockReservation row directly.

inventory.batch.BatchSelectionService — the batch-controlled engine

The single write choke-point for all batch reservation. Every public "save" method (one per source document type) funnels through a private saveSourceLineSelection() that: validates the requested quantity against the document line, resolves FEFO vs. manual, enforces a batch_manual_select permission (bypassed only for a physical scan-pin, since the scan itself is the authorization), releases any existing reservation on that exact line first — re-selection always supersedes rather than adds — pessimistic-locks the candidate BatchMaster rows, re-checks for a race under lock, then writes.

Gap found, not fixed

BatchAllocationStatus.RETURNED and the parentAllocationId field exist and are fully modeled — sumAlreadyReturned() and findReturnableAllocations() are wired to read them — but no call site anywhere in the codebase ever sets status = RETURNED. Either the sales-return batch-reversal flow lives outside this service entirely, or this is unfinished scaffolding.

inventory.reservation.PosStockReservationService — the non-batch counterpart

New (uncommitted at time of writing), deliberately minimal — its own Javadoc calls it a mirror of BatchAllocation "but carries no batch/bin." Two methods: reserveForLayawayLine() and releaseSourceDocument()/releaseLayaway(). No FEFO, no pessimistic locking, no manual-pick modal, no consume/restore pair — because it is scoped to a single caller (POS layaway) and a single unit of granularity (warehouse, not bin or batch).

inventory.warehouse.WarehouseStockService — the read-side aggregator

Where every screen's "Reserved"/"Available" number actually comes from. It branches on product.isBatch(): batch products sum BatchAllocation rows; non-batch products sum three unrelated sources — live Sales Order/Proforma queries, dispatched-not-delivered Delivery Note quantities, and PosStockReservation rows — added together. This three-way sum is duplicated with a variance at the bin level (see Part 11).

Everything else that touches these tables

SalesOrderService

Calls into BatchSelectionService at confirm, save (item-churn cleanup), and delete. Never writes StockMovement itself.

SalesInvoiceService

Owns Fast-Sale auto-FEFO and the POS unified-scan pin path. Releases on SO→INVOICED transition, delete, and cancel.

DeliveryNoteService

The only service that pairs BatchAllocation mutation with StockMovement posting in the same transaction. Owns FEFO deduction for non-batch lines.

PosLayawayService

Calls both reservation services from the same loop; releases both together on every exit path.

BinStockService

A second, narrower aggregator — reserved math for the bin-drill-down screen. Omits PosStockReservation entirely (see Part 11).

StockMovementService

The actual ledger writer — postOutboundStock / reverseOutboundStock. Never reads or writes a reservation table; reservation and movement are cleanly separated services.

Part 3

Database objects

Only one table in the entire schema is dedicated to reservation state. Everything else is either physical stock ledger or live-query aggregation over ordinary document tables.

batch_allocation

Hibernate-generated (ddl-auto=update) — there is no Flyway migration defining this table; the @Entity annotations on BatchAllocation.java are the schema source of truth. No foreign key constraint exists at the database level for batch_master_id or product_id/bin_id — those relationships are enforced only in application code.

ColumnNotes
source_document_type / _id / _line_idNOT NULL. One of SALES_ORDER, SALES_INVOICE, DELIVERY_NOTE, SALES_RETURN (unused), POS_LAYAWAY.
batch_master_id@ManyToOne to the physical batch unit. Each row is effectively one physical unit — quantity is almost always 1.
statusRESERVED · CONSUMED · RELEASED · RETURNED (declared, never written)
allocation_methodAUTO_FEFO or MANUAL. No FIFO value exists — the system models FEFO or explicit override only.
depleted_by_document_type / _id / _line_idSet when a different document (typically a DN) consumes a reservation created by another (typically an SO) — the hand-off trail.
parent_allocation_idReturn-reversal linkage — modeled, unused (see Part 2 gap).

Three indexes: (source_document_type, source_document_id, source_line_id, status), (batch_master_id, status), (product_code, bin_code, status) — all read-path optimized, matching how the service actually queries.

pos_stock_reservations

Defined in V41__pos_stock_reservation.sql — the one reservation table that does have a real migration. Flat and small: source_document_type/_id/_line_id, product_id, warehouse_id (no bin), quantity, status (RESERVED/RELEASED), reserved_by/reserved_at/released_at. No FK constraints, mirroring batch_allocation.

bin_stockCarries a reserved_quantity column that is never written by any service — see Part 11.

Part 4

Inventory quantity logic

The formula is constant everywhere — available = onHand − reserved — but what counts as reserved silently changes depending on product type and which screen is asking.

on-hand (any product)
  = SUM(stock_movement.quantity) WHERE product/warehouse[/bin] matches
    // the only source of physical truth in the system

reserved, batch-controlled product (every screen, consistently)
  = SUM(batch_allocation.quantity) WHERE status = 'RESERVED'
                                      AND product/warehouse[/bin] matches
    // CONSUMED rows excluded — that stock is already gone via StockMovement

reserved, non-batch product — WAREHOUSE level (WarehouseStockService)
  = sales_order_reserved      // live SUM, CONFIRMED/PARTIALLY_PAID/FULLY_PAID, no active DN
  + proforma_reserved         // live SUM, ISSUED, no active DN
  + delivery_note_reserved    // live SUM, status IN (DRAFT, DISPATCHED)
  + pos_stock_reservation     // hard rows, status = RESERVED

reserved, non-batch product — BIN level (BinStockService)
  = sales_order_reserved
  + delivery_note_reserved
    // proforma_reserved and pos_stock_reservation are OMITTED at this level

available = on-hand − reserved   // same formula, different "reserved" input
Confirmed defect

The bin-level formula silently drops two of the four non-batch reservation sources. For a non-batch product with an active POS layaway, warehouse-level "Available" and bin-level "Available" for the same physical stock will disagree by exactly the layaway (and proforma) quantity — and both numbers are one click apart on the same Warehouse Detail page.

Sales Order "reserved" is coarser than it looks

The live query excludes an SO entirely once any non-cancelled Delivery Note exists against it — via NOT EXISTS, a boolean gate, not a partial-quantity subtraction. A sales order with 100 units confirmed and 1 unit dispatched on a DN drops to zero contribution to "reserved," understating held stock for the other 99 units until that DN reaches DELIVERED and the units are physically deducted from on-hand. The window is normally short (dispatch usually precedes delivery closely) but it is a real, reproducible gap, not a hypothetical.

Batch quantity vs. warehouse/bin quantity

Batch-controlled availability is computed at unit granularity — each BatchMaster row is one physical unit, so "batch quantity" is really a row count filtered by status, not a summed decimal. Non-batch quantities are true decimal sums with unit-of-measure conversion applied via ProductPacking.conversion in every JPQL query that touches them (visible in every SO/Proforma/DN reservation query cited above).

Serial quantity

Not part of the reservation graph at all. Serials are marked SOLD at Delivery Note delivery (a status flip on the serial row itself), not reserved ahead of time through any of the mechanisms above.

Part 6

Frontend investigation

Eight screens surface reservation data. One shared modal drives batch selection for three document types; layaway state lives entirely inside one 13,800-line POS page; and the same "Reserved" label means different things depending on which screen you're looking at.

BatchSelectionModal — the shared batch picker

One component, three callers: Sales Order, Delivery Note (read-only when inherited from an upstream document), Sales Invoice. Calls GET /api/inventory/batches/selection-options for the FEFO-vs-manual preview, then one of three document-scoped POST/DELETE .../batch-selection endpoints to commit or clear. Shows available units, days-to-expiry, and a warning badge per row.

Minor finding

The component fabricates a local reserved: true flag on already-saved rows that is never rendered anywhere — vestigial, harmless, but suggests the frontend author once expected a reserved boolean from the API that the backend response never actually carries.

POS Layaway UI

Entirely owned by POSSales.jsx; the touch-screen and console variants are presentational children driven by the same parent state, not separate implementations. Save/Hold, the Layaways list, the detail drawer's "Reserved Items" table, and Convert/Cancel are all wired up.

Confirmed capability gap

The backend exposes POST /api/pos/layaways/{id}/release-reservation (supervisor-gated manual release without cancelling) and both instalment-payment endpoints (POST/GET .../payments) — none of the three have a frontend API wrapper, anywhere. The only stock-release action reachable from the UI is a full Cancel. Two backend capabilities are currently dead code from the frontend's perspective.

Stock Availability Modal

Pure information display — per-location On Hand / Reserved / Available plus incoming LPO quantities, with a footer disclaimer explaining reservations are created at order confirmation and released on expiry/cancellation. No reserve/release action lives here; it only reads.

Warehouse Detail — two reservation numbers, one page apart

The product-level stock table and stat tiles include layaway reservations in their "Reserved" figure. The bin-level drill-down on the exact same page does not — it omits PosStockReservation entirely, matching the backend gap described in Part 4. This is the single clearest example of the architecture's inconsistency being directly visible to an end user.

Stock Transfer, Batch Editor, Transaction Preview

Stock Transfer shows an "Available" figure for transfer purposes that was not confirmed to net out the same SO/DN/Layaway reservations as the Warehouse screens — a follow-up worth a dedicated check before relying on it. The Stock Taking Batch Editor is purely a data-entry tool for batch identity correction and shows no reservation data at all. Transaction Preview (and its Sales Order / Invoice preview siblings) show financial summaries only — no reservation information appears on any read-only preview surface in the app.

Part 9

Current architecture: single system or several?

Three independent mechanisms, not one, coexist with no shared interface or common base type between them.

MechanismScopeGranularityLockingAdvantagesProblems
BatchAllocation Batch-controlled products, all sales-side documents Physical unit (bin + batch) Pessimistic row lock at selection time Mature, well-tested, one write engine for five source types, clean consume/restore pair No FK constraints in the DB; RETURNED status modeled but never used; ownership hand-off (SO→DN) is implicit via sourceLineId lookup, not an explicit transfer
PosStockReservation Non-batch products, POS layaway only Warehouse (no bin, no batch) None — plain save/update Simple, purpose-built, easy to reason about in isolation A second, structurally similar table exists purely because BatchAllocation is hard-wired to require a BatchMaster; the "mirror but simpler" design duplicates the reservation concept instead of generalizing it
Soft / live-query reservation Quotation, Proforma, non-batch Sales Order lines, in-flight Delivery Notes Warehouse, summed live None — no row, no lock, pure read-time aggregation Zero write cost, self-cleaning (aging/status naturally drops rows out of the sum) Boolean SO/DN exclusion instead of partial-quantity subtraction (Part 4); every consumer must independently know which four sources to sum and in what combination — two call sites (warehouse vs. bin) already disagree

The overlap is real but narrow: all three ultimately feed the same available = onHand − reserved formula, and WarehouseStockService is the one place that already knows how to reconcile all three into a single number — it just doesn't do so consistently at every call site (BinStockService reimplements a subset rather than calling through). The coupling is shallow (no shared interface, no common entity), which made it cheap to bolt on PosStockReservation without touching BatchAllocation — but that same shallowness is why the bin-level formula could drift without anything failing loudly. Technical debt is concentrated in exactly one place: the "what counts as reserved for a non-batch product" logic is duplicated in two services instead of owned by one.

Part 11

Current problems

Confirmed, high confidence

  • Bin-level vs. warehouse-level disagreement. BinStockService omits PosStockReservation and Proforma from its reserved sum; WarehouseStockService includes both. Same page, one click apart, different numbers for the same non-batch product.
  • Boolean, not quantity-aware, SO/DN hand-off. The NOT EXISTS gate in SalesOrderRepository.sumReservedQuantityForProductsByWarehouse drops an order's entire reserved contribution to zero the moment any single unit is dispatched, understating held stock for the remaining undelivered quantity.
  • Dead column. bin_stock.reserved_quantity exists in the schema and is set on a response DTO but never persisted to the entity — a trap for any future developer who assumes it's live data.
  • Modeled-but-unused return path. BatchAllocationStatus.RETURNED and parentAllocationId are fully wired for reads (sumAlreadyReturned, findReturnableAllocations) but nothing ever writes them — either dead scaffolding or a return flow that bypasses this service entirely and should be reconciled.
  • Two backend capabilities unreachable from the UI. Supervisor release-without-cancel and layaway instalment recording both exist server-side with no frontend caller — either finish the UI or remove the endpoints; right now they're untested-by-usage surface area.
  • No database-level referential integrity. Neither reservation table has an FK constraint on its target entity (batch, product, warehouse, bin) — correctness depends entirely on application code never producing an orphaned row.

Architectural, moderate confidence

  • Reservation concept duplicated, not generalized. PosStockReservation was built as a parallel-but-simpler mirror of BatchAllocation rather than as a generalization of it (e.g., an optional bin/batch on one reservation table). Every future non-batch reservation need will face the same choice again.
  • "Reserved" has at least three distinct definitions depending on product type and screen — batch sum, four-source non-batch sum, or two-source non-batch sum — with no single function any developer can call to get "the" reserved quantity without knowing which screen they're building.
  • Race conditions are only guarded on the batch path. BatchSelectionService pessimistic-locks; PosStockReservationService does a plain save with no lock — two concurrent layaways on the same non-batch product's last unit could both succeed.
  • Soft reservation on Quotation/Proforma is a warning, not a guard — by design, per the source comments, but worth surfacing explicitly to stakeholders since it means overselling from stacked quotations is possible in principle.

Not investigated further (flagged for follow-up)

  • Whether Stock Transfer's "Available" figure nets out the same SO/DN/Layaway sources as the Warehouse screens — the frontend research pass could not fully pin down which warehouseApi.js call it uses in the time available.

Part 12

Improvement recommendations

Option A — Keep current architecture

Pros: zero migration risk, zero regression surface, batch-controlled reservation (the higher-stakes half, since it covers pessimistic locking and FEFO) is already solid. Cons: the three-definitions-of-reserved problem and the bin/warehouse drift persist and will resurface every time a new screen needs a reservation number. Effort: none beyond the point-fixes below. Risk: low today, compounding over time as more screens are added.

Option B — Unified Reservation Query Layer (read-only aggregation)

Introduce one function — ReservationQueryService.getReserved(productId, warehouseId, binId?) — that internally branches on batch vs. non-batch and sums exactly the sources WarehouseStockService already sums today, then make BinStockService call through it instead of reimplementing a subset. No schema change, no write-path change. Pros: fixes the bin/warehouse drift (Part 11's most visible bug) with a small, low-risk change; establishes one place to fix the SO boolean-exclusion issue later. Cons: doesn't address the underlying duplication between BatchAllocation and PosStockReservation, or the missing locking on the non-batch path. Effort: small — a few days. Risk: low. Backward compatibility: full — it's a refactor of read paths only.

Option C — Central Reservation Service (common reservation engine)

Generalize BatchAllocation into a single reservation table with optional bin_id/batch_master_id, and migrate PosStockReservation rows into it; give every write path (SO, DN, Invoice, Layaway) the same pessimistic-locking, FEFO, and consume/restore semantics regardless of whether the product is batch-controlled. Pros: one mental model, one set of race-condition guarantees, one place to add the RETURNED flow properly. Cons: nontrivial data migration (existing pos_stock_reservations rows need backfilling with nullable bin/batch), touches every call site listed in Part 2's card grid, and the "no FK constraints" debt would need addressing at the same time to be worth doing properly. Effort: large — multi-week, cross-module. Risk: moderate-high, concentrated in the POS layaway non-batch path since it has the least test coverage today. Backward compatibility: breaking at the data layer, requires a migration window.

Recommendation

Recommended

Option B now, Option C only if a concrete new requirement demands it (e.g., bin-level or batch-level POS reservation, which the current PosStockReservation design cannot support without exactly the generalization Option C proposes). The immediate, user-visible bug — bin and warehouse screens disagreeing — is fixed by centralizing the read side, which is low-risk and ships fast. Option C's cost is justified by a specific future need, not by architectural tidiness alone; the two mechanisms today are duplicated but not actually in conflict with each other, and the write-side risk of merging them (particularly around the un-locked non-batch path) is higher than the benefit until there's a concrete reason — such as bin-precise layaway reservation — to take it on.
