File Storage Implementation Blueprint

A critical second pass over File Storage & Multi-Tenant Upload Architecture (2026-07-23). The original correctly diagnosed the problem and the local-disk path convention; this pass tightens the two load-bearing decisions it left loose — tenant identity and database storage format — and adds the metadata, maintenance, security, and DR layers a production rollout needs but the first draft didn't yet specify.

Branch feature/billpos Scope architecture refinement only — no re-audit, no code Supersedes Parts 09–10 of the original study (see Part 11) Prepared 2026-07-23

Part 01

Stable tenant identity

The original study recommended keying the storage path on spring.profiles.active — "no new identity concept, reuse what's already there." That reasoning is right about cost, wrong about durability. This is the one recommendation in the original report that needs to change before implementation, not after.

Why the Spring profile name fails as a permanent storage key

The fourteen profile files already on disk are client1, client2, client4, client5, client6, demo, geebu, hilite, leroyalflowers, leroyalgifts, prod, qa, royaltools, testing. Looking at that list with fresh eyes surfaces three problems the original report didn't weigh:

  • Profile names are not all tenant names. prod, qa, and testing are environment labels, not clients — three of the fourteen profiles don't even represent a distinct customer, or represent one ambiguously. A storage key derived from the profile would put "prod" data in a folder called prod/, which says nothing about which client that is.
  • Profile names are operational shorthand, not identity. client1 through client6 are numbered placeholders assigned in deployment order, not derived from the customer's actual name. If a client is renamed, re-sold, or a profile file is renamed during a later infra cleanup — plausible, since geebu and hilite already show that naming shifted from numbered to name-based at some point — every path on disk and every DB path column keyed on the old profile name goes stale simultaneously.
  • Profile ≠ deployment target 1:1 forever. The whole point of the K8s work already underway is to decouple "which tenant" from "which process/config." Tying the storage key to spring.profiles.active re-couples exactly the thing the K8s migration is trying to make swappable — a new profile naming convention, a profile consolidation, or a future move to one Spring Boot image serving multiple tenants via a request-scoped tenant resolver (already how the K8s ConfigMap pattern half-works today) would all break the storage key.
Correction to the original report

Part 09 of the original study said "no new identity concept is introduced; it's the one tenant key that already exists." That statement undersold the cost: the profile name is a deployment key, not a data key, and conflating the two means a future rename operation (which has already happened once, per the client1..6 → geebu/hilite naming drift visible in the existing profile set) becomes a data migration instead of a config change.

Recommendation: a dedicated, immutable storage identifier

Introduce one new column, tenant_storage_key (or equivalently company.storage_code), set once at tenant provisioning time and never changed again for the life of that tenant — independent of the client's display name, the Spring profile name, the database name, or anything else that might legitimately change over time.

CandidateVerdictReasoning
spring.profiles.activeRejectDeployment/environment label, not data identity; already shows naming drift; conflates 3 non-tenant environments (prod/qa/testing) with real clients.
company.code / company.slugReject as primary key, fine as display valueDerived from the company's trading name — legitimately changes on rebrand, legal name change, or franchise resale. Good for human-readable folder names in exports/reports, bad as the immutable join key.
Database name / JDBC URLRejectChanges on every database migration — exactly the event this whole report exists to make safe. Using it as the storage key would re-couple storage identity to the thing being migrated.
Dedicated tenant_storage_key (UUID or short opaque code, assigned once)RecommendedExists for exactly one purpose: identify this tenant's data forever, independent of every other naming scheme that's allowed to change around it.

Practical shape: a short, human-debuggable code (e.g. tsk-a1x9k2 or a slugified one-time snapshot of the company name, leroyal-flowers-01) rather than a raw UUID — engineers will read this in logs, PVC paths, and tarball filenames often enough that pure opacity costs more than it protects. Store it once, in exactly one place per tenant (recommend the existing per-tenant application-{profile}.properties file, as a new tenant.storage-key=... property — deliberately *not* derived from the profile filename itself, breaking the coupling explicitly rather than by convention).

Migration cost of this correction

Because no tenant has any files namespaced yet (the original report's Part 12 migration hasn't run), this correction costs nothing extra if made now — it only changes what string gets used as the path segment in a migration that hasn't happened. Making this change after Part 12 has already run for even one tenant would mean a second migration. This is the single highest-leverage fix available before implementation starts.

Part 02

Storage abstraction — layered, not a single service

The original report's TenantFileStorageService was directionally correct (one shared component, nine call sites collapse into one) but under-specified as a single concrete class. A production blueprint needs a real interface boundary so the backend can change where bytes live without touching any business logic — matching exactly what Part 06/09 of the original already argued for local-vs-object-storage, just not yet drawn as a layered contract.

Layer design

Controller / Service layer (Product, Brand, Company, Branch, Employee, Receipt, Tax, ...)
        │  calls only this interface — never a concrete class, never java.nio.file directly
        ▼
StorageProvider  (interface)
  - store(StorageKey key, InputStream content, StorageMetadata meta) → StoredFileRef
  - load(StorageKey key) → InputStream
  - delete(StorageKey key) → void
  - exists(StorageKey key) → boolean
  - resolvePublicUrl(StorageKey key) → String
        │
        ▼
  ┌─────────────────────┬──────────────────────────────┐
  │ LocalStorageProvider │  (future, same interface)     │
  │  (default, active)   │  MinIOStorageProvider          │
  │                       │  S3StorageProvider              │
  │                       │  AzureBlobStorageProvider        │
  │                       │  GcsStorageProvider                │
  └─────────────────────┴──────────────────────────────┘

A thin TenantFileStorageService still exists above this — but now it's a tenant-aware convenience wrapper (injects the tenant storage key and category into a StorageKey, delegates to whichever StorageProvider is active) rather than the whole abstraction. The distinction matters: TenantFileStorageService knows about tenants and categories; StorageProvider knows nothing about the business domain at all — it just moves bytes given an opaque key. That separation is what lets the provider be swapped via Spring configuration (storage.provider=local|minio|s3|azure|gcs, a single @ConditionalOnProperty-selected bean) with zero change to any of the nine call sites.

StorageKey — the one shared value type

A small immutable value object, not a raw string, so path construction happens in exactly one place:

record StorageKey(String tenantStorageKey, String category, String filename) {
    String toLogicalPath() { return category + "/" + filename; }          // → §3, DB storage format
    String toPhysicalPath(String root) { return root + "/" + tenantStorageKey + "/" + toLogicalPath(); }
}

Every existing hardcoded directory constant (Part 01/02 of the original report) is deleted; every call site constructs a StorageKey and calls TenantFileStorageService. This is also where filename sanitization and category allow-listing live (Part 08) — centralizing them fixes the Sales Order attachment collision risk the original report flagged (no-UUID original-filename reuse) as a side effect, not a separate task.

Why this layering, specifically, beats a single concrete class

  • Local-to-object-storage swap becomes a deploy-time config change (Part 06/09 of the original correctly wanted this outcome; this is what actually delivers it — a single service class with an if (useS3) branch inside it does not).
  • Testability. Unit tests inject an in-memory StorageProvider fake; today's storage classes require real filesystem I/O in tests (or are untested — the original audit found no dedicated tests for any of the seven storage classes).
  • Hybrid storage (Part 09) becomes expressible: a CompositeStorageProvider that reads from local disk but writes new files to S3, for a gradual live migration, is a straightforward addition to this shape and an awkward retrofit onto a single class.

Part 03

Database design — logical storage key, not a URL

The original report's Part 11 recommended rewriting DB columns to hold a relative URL like /uploads/{tenant-code}/products/{uuid}.jpg. On review, that's the wrong level to freeze into the database — it bakes both the tenant identifier and the URL-serving convention into rows that should outlive both.

Option A — Full URLOption B — Relative pathOption C — Logical storage key
Examplehttps://app.billbull.com/uploads/client1/products/img.jpg/uploads/client1/products/img.jpgproducts/img.jpg
Survives a domain/CDN changeNo — every row needs a rewriteYesYes
Survives a tenant-key renameNoNo — tenant segment is baked into every rowYes — tenant key is supplied at resolve time from the authenticated session/DB connection, never stored per-row
Survives local disk → object storage swapNoNo — /uploads/ prefix is a local-disk-serving artifactYes — the key is provider-agnostic; only resolvePublicUrl() changes
Frontend impactNone (already a full URL)None (original report's assumption)Requires resolving through an API/service call or a lightweight backend URL-signing endpoint rather than reading a ready-made string — a real but bounded cost
CouplingCouples DB to hostname/CDNCouples DB to tenant key and to the local static-resource conventionCouples DB to nothing except "which category, which filename" — the two facts that are actually permanent about a stored file
Correction to the original report

Part 11 of the original study picked Option B specifically because it requires zero frontend change ("the frontend needs zero changes if the new scheme still resolves through the same /uploads/** URL space"). That's a real, valid short-term argument — and for the immediate migration described in the original Part 12, it is the pragmatic choice. But it optimizes for the frontend cost of this one migration at the expense of baking two more assumptions (tenant key, serving convention) into ~10 database columns that Part 09 of the original report already predicted would need to survive a future storage-backend swap. Section §10 below reconciles this: use Option C as the stored value, and add exactly one new resolution step (already required anyway by the StorageProvider abstraction in Part 02) so the frontend-simplicity goal is still met without re-baking assumptions into data.

Recommended: Option C, resolved through the storage abstraction

Store products/{uuid}.jpg — category and filename only, no tenant segment, no protocol, no /uploads/ prefix. At read time, the same StorageProvider.resolvePublicUrl(StorageKey) from Part 02 combines the logical key with the current request's tenant (resolved the normal way — the active DB connection already implies the tenant, exactly as every other tenant-scoped query in the app already works) to produce whatever URL is currently correct — local /uploads/{tenant-key}/products/{uuid}.jpg today, a signed S3 URL tomorrow, without ever touching the stored row.

This is a genuine change from the original report's Part 10 assumption of "zero frontend changes." The honest accounting: getImageUrl() in urlUtils.js currently assumes the backend already returns a ready-to-use path. Under Option C, the backend DTO layer (not the frontend) does one extra step — resolving the logical key to a URL before serializing the response — so the frontend's getImageUrl() contract is actually preserved unchanged; only backend serialization gains one resolution call per entity, already inside the existing DTO-mapping code, not a new endpoint round-trip.

Part 04

Export / Import client architecture

The original report's Part 08 (client migration) already gets 80% of the way here — tenant-code-rooted folders make a tarball-per-tenant trivial. This section generalizes that into a named, repeatable capability rather than a one-off migration script, and folds in the assets the original scope named but didn't structurally account for: company configuration, templates, documents.

Export package contents

{tenant-storage-key}-export-{date}/
  manifest.json              ← export tool version, timestamp, tenant-storage-key, checksums of every file below
  database.sql.gz            ← pg_dump of this tenant's database only
  uploads/
    products/
    brands/
    company/
    branch/
    employees/
    avatars/
    sales-orders/
    receipts/
    tax_filings/
  config/
    company-profile.json     ← non-file company settings not captured by the DB dump alone, if any live outside it
    print-templates/         ← from src/utils print template config, if template bodies are DB-stored, exported alongside
  checksums.sha256           ← one line per file, verified on import (§7 duplicate/orphan detection reuses this same hashing)

manifest.json is the piece the original report didn't need (it designed a one-time migration, not a repeatable export) but a real Export/Import feature does: a self-describing header so an import tool can validate compatibility (schema version, storage-key format version) before touching anything.

Export flow

1. Resolve tenant storage key (Part 01) — the one stable value the whole package is keyed on
↓
2. pg_dump that tenant's database (already fully independent per-tenant, per the existing "one DB per tenant" model — no change needed here)
↓
3. Archive uploads/{tenant-storage-key}/ via the active StorageProvider — local disk today (a tar), object storage tomorrow (a prefix-scoped bucket export) — the export tool calls the abstraction, never the filesystem directly, so it works unmodified after any future storage backend swap
↓
4. Compute checksums, write manifest.json, bundle into one package

Import flow

The inverse, plus one safety step the original migration plan didn't need but a general-purpose import tool must have: collision detection. Importing into an environment that already has a tenant using the same storage key must fail loudly, not overwrite — this is the same "never destructive without confirmation" principle applied to tenant boundaries instead of files.

  • Validate manifest compatibility (schema/format versions) before any write.
  • Restore database into a freshly named target DB (never overwrite an existing DB with dump restore — matches standard practice already implied by the existing per-tenant-DB model).
  • Extract uploads into a new uploads/{tenant-storage-key}/ (or a caller-supplied new key, for cloning a tenant into a sandbox/staging copy — a capability this design gets for free once the storage key is decoupled from deployment identity, per Part 01).
  • Verify every file against checksums.sha256 before declaring success — reuses the same hashing infrastructure recommended for orphan/duplicate detection in Part 07, not a separate mechanism.
Why the earlier design already supports this "for free"

Because Part 01 decouples the storage key from the Spring profile, and Part 02's abstraction means export/import code calls StorageProvider rather than raw paths, Export/Import isn't a redesign of the upload system — it's a new tool built entirely on top of the same two abstractions everything else already uses. This is the direct payoff of getting Parts 01–02 right before writing any of the nine call-site refactors.

Part 05

Backup & disaster recovery

The original report treated backup/DR as an existing-but-unaudited external concern ("not found as existing automation in this repo, likely a manual/host-level process today"). This section turns that gap into an explicit design rather than leaving it implicit.

Independent, per-tenant, per-asset-class backups

AssetBackup mechanismIndependent of other tenants?Independent of the other asset class?
Databasepg_dump per tenant DB, scheduledYes — already true todayYes — restorable without touching uploads
UploadsTarball/snapshot of uploads/{tenant-storage-key}/ only, scheduledYes — only after Part 01's fix; today's flat directory makes this impossible without manual filtering, exactly the original report's core findingYes — restorable without touching the DB
Company config / templatesIncluded in the DB dump if DB-stored (confirm during implementation which print-template assets are DB rows vs. files); otherwise treated as an uploads-category and covered by the same snapshotYes, once classifiedn/a

Retention policy

Recommend a standard grandfather-father-son rotation, independently configurable per asset class since DB dumps and upload snapshots have different size/change profiles: daily backups retained 14 days, weekly retained 8 weeks, monthly retained 12 months — tunable per tenant tier if some clients contractually require longer retention. This is a policy recommendation, not yet implemented tooling; the existing `audit.retention.months` pattern already used for audit logs is a reasonable model to extend to backups.

Restore verification — don't trust a backup you haven't restored

A backup that has never been restored is a hope, not a guarantee. Recommend a scheduled, automated "restore drill": periodically (e.g. monthly) restore the latest backup of one rotating tenant into an isolated scratch environment and run the same validation checklist the original report's Part 13 already defined for the migration (HTTP-fetch every path, row-count sanity checks) — reusing that checklist as ongoing DR hygiene, not a one-time migration gate.

Disaster recovery workflow

1. Declare incident; identify last-known-good backup timestamp for the affected tenant(s)
↓
2. Provision a clean target (new PVC / new DB instance) — never restore over a partially-corrupted live target
↓
3. Restore DB dump and upload snapshot independently and in parallel (they don't depend on each other, per the table above)
↓
4. Run the validation checklist (original report Part 13) before cutting traffic back
↓
5. Cut over; keep the failed environment quarantined (not deleted) until root cause is understood

Because uploads and database are independently backed up and independently restorable, a corruption event affecting only one (e.g. a bad migration script mangling DB rows, or a disk failure on the uploads PVC) never forces restoring the other — directly satisfying the task's requirement that "uploads and databases remain independently restorable."

Part 06

File metadata — centralize it

The original report didn't address metadata at all beyond the bare path/filename columns already scattered across ten entities. On review, this is a real gap: none of the nine existing storage call sites record MIME type, size, checksum, or uploader — all facts that matter for the maintenance (Part 07), security (Part 08), and export/DR (Parts 04–05) capabilities this blueprint is adding.

Recommendation: yes, centralize — one new table

Add a single stored_file table, written once per StorageProvider.store() call (i.e., inside the shared TenantFileStorageService, not duplicated per module — the same collapsing-nine-into-one principle as Part 02):

ColumnPurpose
idPK (UUIDv7, matching the project's existing BaseEntity convention)
tenant_storage_keyWhich tenant, per Part 01 — redundant with the DB itself being per-tenant, but valuable for any future cross-tenant admin/reporting tooling and for the export manifest in Part 04
logical_keyThe Option C value from Part 03 (products/{uuid}.jpg) — the join key other entities' path columns reference
original_filenameAs uploaded by the user — needed for download-as, and for the collision/sanitization audit in Part 08
stored_filenameThe generated (UUID-based) name actually on disk/in the bucket
mime_typeDetected server-side (Part 08 — never trust the client-supplied Content-Type alone)
size_bytesFor storage usage reports (Part 07) and max-size enforcement audit trail
checksum_sha256Duplicate detection (Part 07), integrity verification on export/import (Part 04), corruption detection during DR restore (Part 05)
storage_providerlocal / s3 / minio / azure / gcs — makes hybrid storage (Part 09) queryable: "show me every file still on local disk"
categoryproducts, brands, company, etc. — the same category taxonomy already established
uploaded_by, created_atStandard BaseEntity audit fields, already the project convention

The nine existing entity columns (Product.imageUrl, Brand.logoPath, etc.) keep their current role as the domain-facing reference — they store the same logical_key value (Part 03), not a foreign key to stored_file.id, to avoid a disruptive join-everywhere refactor. stored_file is a secondary index over the same logical keys, populated as a side effect of every write, useful for cross-cutting queries (usage reports, orphan detection, duplicate detection) that no single domain entity can answer on its own today.

Cost/benefit, stated plainly

This is one new table and one extra insert per upload — cheap. The alternative (no centralized metadata) means every future maintenance feature in Part 07 has to independently union-query nine different entity tables to answer "what files exist and how big are they," which is exactly the kind of duplicated-logic problem this whole redesign exists to stop introducing.

Part 07

Storage maintenance

None of this exists today (the original audit found no cleanup jobs, no usage reporting for uploads) and none of it was in scope for the original report. All of it becomes straightforward once Part 06's stored_file table exists — each feature below is a query or a scheduled job over that one table, not new per-module logic.

Orphan file detection

A file exists in stored_file (or on disk) with no domain entity referencing its logical_key — e.g. a Product was deleted but its image row/file was not cleaned up. Scheduled job: for each category, join stored_file against the owning entity table's active rows; anything unmatched is a candidate orphan.

Automatic cleanup

Orphans older than a grace period (e.g. 30 days — long enough to survive soft-delete/undo windows, matching the project's existing isActive soft-delete convention) are safe to physically delete. Never delete on first detection; always a report-then-sweep cadence, mirroring the project's existing AuditLogRetentionJob pattern.

Duplicate detection

checksum_sha256 makes this a single GROUP BY checksum_sha256 HAVING COUNT(*) > 1 query per tenant. Useful for storage cost reports; actual dedup (content-addressed storage) is a further optimization, not required for v1.

Storage usage reports

SUM(size_bytes) GROUP BY tenant_storage_key, category — direct, cheap, and finally possible because size is actually recorded (it isn't today).

Per-tenant storage statistics

Same query, surfaced as an admin-facing dashboard tile — total storage, growth trend, largest category — useful both operationally and as a factual basis for any future tiered-pricing-by-storage conversation.

Folder health validation

A scheduled check that every tenant_storage_key with active DB rows actually has its expected category subfolders present on the active StorageProvider — catches a mis-mounted PVC or a failed export/import early, before a user hits a 404.

Part 08

Security review

The original report didn't do a security pass on the upload paths at all. This one matters independently of the tenant-isolation redesign — several of these gaps are exploitable today, in the current flat-directory system, regardless of whether this migration ships.

Part 09

Part 10

Final implementation blueprint

Final recommended architecture

Local disk today, behind a StorageProvider interface (Part 02), keyed on a dedicated immutable tenant_storage_key (Part 01, not the Spring profile), with database columns holding a provider-agnostic logical key (Part 03, not a URL), a centralized stored_file metadata table (Part 06) backing maintenance/security/export tooling (Parts 04, 05, 07, 08), and object storage documented as a config-only swap-in for whenever horizontal backend scaling is actually needed (Part 09) — not adopted now.

Folder structure (on the active LocalStorageProvider)

uploads/
  {tenant-storage-key}/          ← e.g. tsk-a1x9k2, NOT the Spring profile name
    products/
    brands/
    company/
    branch/
    employees/
    avatars/
    sales-orders/{orderId}/
    receipts/
    tax_filings/

Structurally identical to the original report's Part 10 — the only change is what fills in {tenant-storage-key}: a dedicated stable identifier (Part 01) instead of the Spring profile name.

Storage abstraction design

StorageProvider interface → LocalStorageProvider (active) → future MinIOStorageProvider / S3StorageProvider / AzureBlobStorageProvider / GcsStorageProvider, selected by config, wrapped by a tenant-aware TenantFileStorageService that all business logic calls (Part 02). StorageKey is the shared value type that centralizes path construction, sanitization, and traversal checks (Parts 02, 08).

Tenant identification strategy

A dedicated, immutable tenant_storage_key, assigned once at provisioning, stored in that tenant's own properties file as a new property (never derived from the filename or the Spring profile name itself). Survives client renames, profile renames, environment consolidation, and any future move away from one-profile-per-tenant deployment (Part 01).

Database storage strategy

Logical storage key only (e.g. products/{uuid}.jpg) in every existing path column — no tenant segment, no protocol, no serving-convention prefix baked in. Resolved to an actual URL at serialization time via StorageProvider.resolvePublicUrl(), keeping the frontend's existing getImageUrl() contract intact (Part 03).

Migration strategy

The original report's Part 12 (copy → rewrite DB → backward-compatible fallback route → validate → cutover) remains structurally sound and is not being replaced — only its inputs change: the path segment inserted is tenant_storage_key rather than the Spring profile name, and the DB rewrite target is the Option C logical key rather than a full relative URL. Because no tenant has been migrated yet, this costs nothing extra to correct now.

Backup strategy

Independent per-tenant, per-asset-class backups (DB dump, upload snapshot) on a grandfather-father-son retention schedule, with scheduled restore-drill verification reusing the same validation checklist as migration cutover (Part 05).

Security architecture

Centralized in the shared storage layer: traversal-safe key construction, generated-only stored filenames, server-side MIME sniffing, per-category extension allow-lists, per-category size caps, executable-signature deny-listing, locked-down response headers — with a clean future insertion point for antivirus scanning as a provider decorator (Part 08).

Kubernetes compatibility

No structural change to the existing per-tenant PVC / Recreate-strategy / single-replica model (original report Part 07 stands). The tenant_storage_key subfolder lives inside the same mount point. The ReadWriteOnce constraint is explicitly *not* solved by this blueprint — it's solved later, by the object-storage swap-in described in Part 09, when and if multi-replica scaling is actually needed.

Future cloud migration path

Config-only StorageProvider swap (local → MinIO/S3/Azure/GCS), CDN URL resolution, ReadWriteMany or object storage removing the single-replica constraint, hybrid per-tenant provider routing — all reachable without touching business logic, per Part 09's table.

Risks

  • Scope creep risk: this blueprint adds a new table (stored_file), a new interface layer, and a new identity field on top of the original's simpler nine-call-site refactor. Recommend treating Parts 01–03 (tenant key, storage abstraction, logical DB key) as the non-negotiable core of Phase 1, and Parts 04–08 (export/import, DR automation, maintenance jobs, security hardening) as separately schedulable follow-on phases — see the phase table below.
  • Frontend resolution cost: Option C (Part 03) requires the backend to resolve logical keys to URLs at serialization time — a small, real addition to every DTO mapper that currently returns a path column verbatim. Bounded and mechanical, but not literally "zero change" the way the original report's Option B was.
  • Tenant-key provisioning discipline: the whole benefit of Part 01 depends on the storage key never being reused or reassigned. This needs to be a documented provisioning-checklist item, not just a good intention — a single accidental key collision between two tenants would recreate the exact cross-tenant exposure this report exists to prevent.

Trade-offs

  • Logical storage key (Option C) trades a small backend resolution cost for permanent independence from hostname, tenant-key, and storage-backend changes — judged worth it given this report's stated purpose is specifically to survive exactly those kinds of changes.
  • A dedicated stored_file metadata table trades one extra insert per upload for making Parts 04, 05, 07, and 08 straightforward queries instead of ad-hoc per-module logic — judged worth it since those four capabilities were explicitly requested.
  • Staying on local disk (not adopting object storage now) trades the CDN/horizontal-scaling benefits of Option C from the original report for zero new infrastructure risk today — judged worth it because nothing in the current single-replica-per-tenant K8s model needs those benefits yet, and the abstraction preserves the option cleanly for later.

Which architecture is the final, production-ready one

Verdict

This document, File Storage Implementation Blueprint, is the final architecture to implement against — not because the original report was wrong in its overall direction (it correctly diagnosed the problem, correctly ruled out premature object-storage adoption, and correctly designed the folder/migration shape), but because two of its foundational choices — tenant identity source and database storage format — would have been expensive to unwind after Phase 1 shipped, and this review catches both while the cost of changing them is still zero. Treat the original report as the accurate problem statement and audit (Parts 01–08 of that document remain valid ground truth and are not superseded), and this blueprint as the implementation-ready design that supersedes its Parts 09–14 specifically for the tenant-key and database-format decisions, while carrying forward everything else unchanged.

Prepared as a critical architecture review only — no code was written or changed, no existing report's findings were re-audited beyond confirming that no tenant.code/company.code-equivalent field or object-storage dependency already exists (both confirmed absent). All recommendations here are design decisions, not verified-against-code claims, except where explicitly cross-referenced to the original 2026-07-23 audit.
