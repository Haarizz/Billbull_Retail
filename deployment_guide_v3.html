1. Backend Architecture

BillBull uses a single Spring Boot backend codebase. One JAR is shared by every client; each client runs as a separate Java process bound to its own profile, database, and port.

billbull-backend/
|-- application.properties              # Common configuration
|-- application-client1.properties      # Nest
|-- application-client2.properties      # Max Eagle
|-- application-hilite.properties       # Hilite
|-- application-client4.properties      # Al Badar
|-- application-client5.properties      # AGI
|-- application-client6.properties      # Helenz
|-- application-qa.properties           # QA
|-- application-testing.properties      # Testing
|-- application-geebu.properties        # Geebu

Each profile contains:

  • Database connection
  • Server port
  • Client-specific configurations

The Download PDF feature (Quotation, Invoice, etc.) renders documents with headless Chromium driven by Playwright on the backend, producing vector, selectable-text PDFs identical to the on-screen print preview. This requires a one-time setup on the VPS — see Section 2.

2. One-Time VPS Prerequisites

Run these once per fresh server. Skip any tool already installed.

2.1 Core tooling (Java 17, Maven, Node, Nginx, Certbot)

apt update
apt install -y openjdk-17-jdk maven nginx certbot python3-certbot-nginx
# Node.js 20 LTS (for the frontend build)
curl -fsSL https://deb.nodesource.com/setup_20.x | bash -
apt install -y nodejs
# Verify
java -version
mvn -version
node -v
npm -v

2.2 PDF engine — Chromium for Playwright (REQUIRED for Download PDF)

Why this step is needed
The backend downloads the Chromium binary by itself on first use, but it cannot install Chromium's Linux system libraries (fonts, libnss3, libatk…) because that needs root + apt. This one command installs both, into a shared location all client processes use.

Install Chromium and its system libraries into the shared path /opt/ms-playwright:

cd "/root/BB/Billbull ERP/billbull-backend"
PLAYWRIGHT_BROWSERS_PATH=/opt/ms-playwright \
  mvn exec:java \
    -Dexec.mainClass=com.microsoft.playwright.CLI \
    -Dexec.args="install --with-deps chromium"

Verify the browser landed in the shared path (expect a chromium-XXXX folder):

ls -la /opt/ms-playwright
Re-run this command only when: setting up a new VPS, or after the Playwright version is upgraded in pom.xml. A normal JAR redeploy does not need it — Chromium is already on disk and shared.
Self-healing fallback: if you forget this step, the first Download PDF click makes the backend auto-download the Chromium binary into /opt/ms-playwright on its own (system libraries from --with-deps still need root once). If the binary is present but the system libraries are missing, the app log prints the exact --with-deps command to run.

8. Backend JAR Replacement During Deployment

Go to backend directory:

cd "/root/BB/Billbull ERP/billbull-backend"

Remove old build:

rm -rf target/*

Create new build:

mvn clean package -DskipTests

Restart all profiles using the commands from Section 5 (keep the PLAYWRIGHT_BROWSERS_PATH prefix).

A JAR redeploy does not re-download Chromium — it is already shared in /opt/ms-playwright. Only re-run Section 2.2 if you bumped the Playwright version.

Install dependencies:

npm run build

10. Nginx Management

Edit configuration:

sudo nano /etc/nginx/sites-available/billbull

Test Nginx:

sudo nginx -t

Reload Nginx:

sudo systemctl reload nginx

Restart Nginx (only if required):

sudo systemctl restart nginx
PDF uploads / large requests: the Download PDF request POSTs the document HTML to the backend. If a very large document is ever rejected with 413, raise the Nginx body limit in the server block: client_max_body_size 25m; then sudo nginx -t && sudo systemctl reload nginx.

13. PDF Engine — How It Works & Troubleshooting

How it works

  • The frontend builds the document's print HTML and POSTs it to POST /api/documents/pdf (authenticated by the user's JWT).
  • The backend renders that HTML with headless Chromium (Playwright) — the same engine as the browser print preview — so page breaks, @page rules and table headers are correct.
  • A single Chromium is launched lazily on first use and reused across requests; all client processes share /opt/ms-playwright.
  • If Chromium is missing, the backend auto-downloads the binary once (guarded by a lock so concurrent processes don't race).

Quick diagnostics

Was Chromium launched?

grep -i "headless Chromium" *.log

List the shared browser install:

ls -la /opt/ms-playwright

Manually re-run the full install (binary + system libs) as root:

cd "/root/BB/Billbull ERP/billbull-backend"
PLAYWRIGHT_BROWSERS_PATH=/opt/ms-playwright \
  mvn exec:java -Dexec.mainClass=com.microsoft.playwright.CLI \
  -Dexec.args="install --with-deps chromium"
Symptom in logCauseFix
"Executable doesn't exist" / "please run install" Chromium binary missing Run Section 2.2 (or let first PDF click auto-download it; needs internet)
"error while loading shared libraries" / "libnss3" / fontconfig errors System libraries missing Run the --with-deps command as root (Section 2.2)
Each process downloads its own Chromium PLAYWRIGHT_BROWSERS_PATH not set / inconsistent Add the prefix to every start command (Section 5)
PDF request fails with 413 Nginx body limit too small Set client_max_body_size 25m; and reload Nginx
Download hangs on a fresh server, first time only One-time Chromium download in progress Wait for it to finish; subsequent PDFs are instant
Air-gapped / no outbound internet?
Both the install command and auto-install need access to playwright.azureedge.net. On a locked-down VPS, run the install on a machine with internet using the same Playwright version, then copy the resulting ~/.cache/ms-playwright (or /opt/ms-playwright) folder to the VPS at /opt/ms-playwright, and still install the system libraries with apt manually.

14. Important Notes

  • Do not rename application-client1.properties, application-client2.properties, etc.
  • Domain names and Spring profiles are independent.
  • Each profile contains its own database and server port configuration.
  • Nginx handles subdomain routing to the correct backend.
  • All clients share the same backend JAR but run as separate Java processes.
  • All clients also share one Chromium install at /opt/ms-playwright for PDF rendering.
  • SSL certificates are domain-specific and managed by Certbot.
  • Always check application logs after deployment.
  • The one-time Chromium prerequisite (Section 2.2) is needed only on a fresh VPS or after a Playwright version bump — not on routine JAR redeploys.
