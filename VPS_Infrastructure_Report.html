⚠ Critical risk
  • The system runs with essentially no memory headroom — under 150 MB free/available at all times.
  • No -Xmx heap cap is set on any of the 8 JVMs. Each service's heap can still grow further under real request load; today's RSS figures are near-idle baseline, not peak.
  • No swap exists. Swap is normally the buffer that turns a memory spike into slow-but-survivable degradation. Without it, any spike goes straight to the kernel's OOM killer.
  • The OOM killer does not necessarily kill the process causing the spike — it scores all processes and kills whichever looks most "killable." In practice, one tenant's traffic surge can silently take down an unrelated tenant's backend, producing a 502 Bad Gateway with no direct log correlation to the real cause.
  • Only 2 CPU cores are shared across 8 JVMs, Nginx, and Postgres — a secondary contention risk under concurrent multi-tenant load.
  • This already happened today — this box produced multiple 502 errors purely from services being memory/CPU-starved during startup, before any real user traffic was involved.

Key finding

Rebalancing tenants between boxes is not a viable fix. Moving a tenant from VPS 1 to VPS 2 doesn't solve anything — VPS 2 has proportionally similar RAM-per-tenant economics (3.8 GB / 4 ≈ 950 MB vs. 7.8 GB / 8 ≈ 975 MB) and only one CPU core, so it would just relocate the same risk, not eliminate it.

Recommendations, in priority order

  1. Add swap on both boxes immediately. Cheapest, zero-risk, no app changes required. Even 2–4 GB of swap converts an instant hard-crash into slower-but-survivable behavior, buying time for the real fix. Both boxes have ample free disk (86 GB and 36 GB available respectively) to spare for this.
  2. Cap -Xmx per JVM on both boxes. Prevents any single tenant's heap from growing unbounded and starving its neighbors. Based on current usage, a starting cap of ~1.2–1.5 GB on VPS 1's tenants and ~800 MB–1 GB on VPS 2's tenants gives headroom for real traffic without allowing runaway growth. This constrains blast radius but does not add capacity.
  3. RAM upgrade is the real long-term fix, especially for VPS 1. At 8 tenants already exceeding 7.8 GB at idle, this box needs meaningfully more RAM (16 GB would give ~2 GB/tenant with real headroom) before adding further clients or expecting stable concurrent multi-user traffic.
  4. VPS 2 needs a CPU upgrade more than RAM given it's down to a single core across 4 JVMs + Postgres + Nginx — 2 cores minimum if more tenants are planned there.
  5. Patch and reboot VPS 2. 199 days uptime, 72 pending updates (16 security), and a flagged "system restart required" — schedule a maintenance window. A clean reboot after patching is also a good moment to add swap and heap caps in the same pass.
  6. Treat tenant onboarding as a capacity-planning decision going forward. The current pattern (~900 MB–1 GB RAM per tenant, unbounded) means each new client added to an existing box directly reduces headroom for every other tenant on it.
